本文记录 bilirec 在低配设备上的实测表现，便于你评估是否适合自己的部署环境。

## 测试环境

- 设备：树莓派 5B 16GB
- 容器：Docker（1 GB 内存限制）
- 场景：默认配置（`DROP_FILE_PAGE_CACHE=true`、`LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=60`），每路直播 1080p，同时录制多个直播间
- 指标：CPU 峰值占用；**内存峰值** = 容器监控显示的内存用量（默认配置下更接近 bilirec 实际占用）

<Aside type="note">
以下数据为实测快照，改配置或版本后会有偏差。**给容器分配内存上限**时，请仍用 [内存占用估算](/zh-cn/configuration/memory-estimation/)（按缓冲配置估算，偏保守，通常高于监控表）。
</Aside>

## 数据汇总

| 并发路数 | CPU 峰值占用 | 内存峰值占用 |
|---------|------------|-------------|
| 初始闲置 | ~0.0% | ~7 MB |
| 1 路并发 | ~2.1% | ~65 MB |
| 2 路并发 | ~2.6% | ~86 MB |
| 3 路并发 | ~3.6% | ~103 MB |
| 4 路并发 | ~4.1% | ~144 MB |
| 5 路并发 | ~4.9% | ~173 MB |
| 恢复闲置 | ~0.0% | ~54 MB |

<Aside type="note">
**1～3 路、5 路**为实测（约 65 / 86 / 103 / 165 MB）；**4 路 ~134 MB** 按 3 路与 5 路线性插值。若关闭冷缓存释放，长录时 Docker 监控可能因系统缓存录制文件而远高于上表（单路长录曾顶近 1 GB），并不代表 bilirec 本身多占了那么多。见 [常见问题 · 监控内存偏高](/zh-cn/guides/faq/#性能与容量规划)。
</Aside>

<Aside type="note">
停录后内存会比刚启动时高一些，是因为程序会保留一部分缓冲供下次录制复用（见上表「恢复闲置」），属于正常现象。**勿仅凭本表数字设置 Docker 内存上限**——请用 [内存占用估算](/zh-cn/configuration/memory-estimation/)。
</Aside>

<Aside type="tip">
**常见问题：** 如何用实测数据评估自己的设备、停录后内存为何仍偏高——见 [常见问题](/zh-cn/guides/faq/#性能与容量规划)。
</Aside>
