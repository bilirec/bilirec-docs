遇到报错时，先看界面或日志里的那句中文；若已经说清楚了，按提示操作即可。下面只整理**容易误会、或界面上看不出来原因**的情况。

## 后端录制

### 开了内网穿透和 HTTPS 后，程序启动就报错？

**你会看到：** 一启动就退出，日志里提到 FRP 或 HTTPS 配置不对。

**为什么：** 内网穿透连回本机的协议，必须和你本机是否开 HTTPS 一致。两边开关对不上，程序会拒绝启动，避免连上却用不了。

**你可以：**
1. 若本机**没有**配证书，内网穿透用 HTTP 方式连回来（常见家用场景）。
2. 若本机**已经**配了证书走 HTTPS，内网穿透也要用 HTTPS 方式连回来。
3. 具体变量组合见 [远程访问（FRP）](/zh-cn/configuration/frp/)。

### FRP 服务器地址填错了？

**你会看到：** 日志里写「无效的 FRP_SERVER 格式」，或隧道一直连不上。

**为什么：** 地址一般应写成 `主机名或IP:端口`，例如 `tunnel.bilirec.org:7000`。只写域名时，程序会默认用 7000 端口；不要写成带 `http://` 的网址。

**你可以：** 改成 `地址:端口` 后重启；自架 FRP 时还要填好 Token（见下一条）。

### 用自己的 FRP 服务器，外网还是进不来？

**你会看到：** 程序能启动，但日志里 FRP 初始化失败，或没有公网地址。

**为什么：** 连**自己的** FRP 时通常要填认证 Token；用官方公共服务时，只开 `FRP_ENABLED=true` 即可。

**你可以：**
1. 确认 `FRP_SERVER`、`FRP_BASE_DOMAIN` 与服务器一致。
2. 自架服务填上 `FRP_TOKEN`。
3. 详见 [远程访问（FRP）](/zh-cn/configuration/frp/)。

### 同一场直播里为什么会出现「标题-1」「标题-2」多个文件？

**你会看到：** 目录里除了 `标题-时间戳.flv`，还有 `标题-时间戳-1.flv`、`标题-时间戳-2.flv` 等同一场、**同一时间戳**下的分段。

**为什么：** 这是**直播轮转**（例如 PK、换画质导致流头变化）时程序自动切的新文件，属于正常设计，不是坏了。

**你可以：** 按文件名顺序观看即可。若你想了解轮转规则，见 [录制指南](/zh-cn/guides/recording/)。

### 断网重连后怎么又多了一个不同时间戳的文件？

**你会看到：** 网络卡了一下之后，目录里多了一个**新的时间戳**文件（文件名里没有 `-1`、`-2` 这种后缀）。

**为什么：** 断线后程序会重连并**新开一个文件**继续录，避免把两段不连续的流硬拼在一起。这和上一节说的「同时间戳带 -1/-2」不是一回事。

**你可以：** 按时间戳区分各段即可；若经常断网，先检查网络。重连相关配置见 [录制指南](/zh-cn/guides/recording/)。

### 开了「录完转 MP4」，却一直不转？

**你会看到：** 录制结束了，列表里仍是 flv/ts，或转码任务很久不动。

**为什么：** 默认为了省 CPU，**正在录别的房间时**不会跑本地转码；机器上也可能没装 ffmpeg，或没配云端转码。

**你可以：**
1. 等所有录制都停掉后再看是否开始转。
2. 确认已安装 ffmpeg，或配置了 CloudConvert。
3. 若要在录制时也转，需在配置里单独打开（见 [转换配置](/zh-cn/configuration/conversion/)）。

### 转码老是失败、反复重试？

**你会看到：** 转换任务一会儿失败一会儿又排队。

**为什么：** 常见原因有两类：一是转出来的 MP4 **太小**（不到 1MB 或不到原文件一半），程序会认为结果不可靠而重试；二是**源文件本身不完整或已损坏**（录制中断、磁盘问题等），怎么转都难成功。

**你可以：** 先确认源 flv/ts 能否正常播放；源文件有问题时只能保留原格式或重新录。详见 [转换配置](/zh-cn/configuration/conversion/)。

### 大文件走 CloudConvert 创建任务失败？

**你会看到：** 日志里和「公开地址」「预签名」有关的错误。

**为什么：** 云端要回来拉你的文件，需要填一个**能从外网访问到的完整网址**（`PUBLIC_BASE_URL`），不能留空或只写内网地址。

**你可以：** 在配置里设好对外的 `http://` 或 `https://` 地址，或改用本机 ffmpeg 转小文件。见 [转换配置](/zh-cn/configuration/conversion/)。

## Web 管理界面

### 打开 app.bilirec.org，却连不上自己的录制服务？

**你会看到：** 页面能开，但登录或列表一直失败。

**为什么：** `app.bilirec.org` 是 **HTTPS 网页**，不是你的录制程序本身。网页要单独填后端地址才能连上。由于浏览器安全限制，从 HTTPS 页面去连 **HTTP 局域网地址**（如 `http://192.168.x.x:8080`）往往会被拦下；**HTTP 后端一般只能填本机** `http://localhost:8080` 或 `http://127.0.0.1:8080`。

**你可以：**
1. 后端跑在本机时，用 `http://localhost:8080` 或 `http://127.0.0.1:8080`。
2. 后端在局域网另一台机器上时，需要给后端配 **HTTPS**，或在同一台机器上用浏览器打开管理页，不能指望 HTTPS 的 app.bilirec.org 直接连家里的 HTTP 地址。

### 收不到开播推送通知？

**你会看到：** 浏览器或 PWA 里从没弹过通知。

**为什么：** 需要允许通知权限、成功连过后端，并且用**已保存服务器地址**的方式打开过管理页；只读账号也不会推送。若用的是 **PWA 或 Chrome**，在内地网络下还可能因为**连不上 Google FCM** 而收不到推送（与 bilirec 配置无关）。

**你可以：**
1. 在浏览器设置里允许本站通知。
2. 确认登录页的服务器地址正确并已保存。
3. 用管理员账号订阅推送；内地使用 PWA 或 Chrome 的用户若仍无推送，可改用 SSE 等方案。详见 [Web 界面](/zh-cn/guides/web-ui/) 与 [通知配置](/zh-cn/configuration/notifications/)。

## Android

### 录制保存到 SD 卡后变慢？

**你会看到：** 换到 SD 卡路径后，录制或界面变卡。

**为什么：** 外置 SD 通常比手机内置存储慢。是否在 SD 上启用「排队写盘」等策略，请在 **bilirec-mobile App 设置**里调整（存储策略 / 序列化写入），不再由后端自动改。

**你可以：** 在 App 的存储相关设置里按需开关；追求速度可改回内置存储。见 [Android 指南](/zh-cn/guides/android/) 与 [Android 默认配置](/zh-cn/configuration/android-defaults/)。

### 启动后切到后台，过一会儿服务就自己停了？

**你会看到：** 刚还能录，划走 App 或锁屏一会儿，通知栏服务没了或 Web 连不上。

**为什么：** 最常见是**没关电池优化**（或系统杀后台）。Android 对后台和省电管得很严，不按提示放行就容易被停掉。

**你可以：**
1. 按 App 提示**关闭电池优化**、允许通知（前台服务需要）。
2. 若选了外置存储路径，按提示授予存储权限。
3. 尽量别让系统「清理后台」。见 [Android 指南](/zh-cn/guides/android/)。

### 用 bilirec-mobile 打开 PWA 后，连不上本机后端？

**你会看到：** 默认地址连不上，或换浏览器才好使。

**为什么：** `localhost` 和 `127.0.0.1` **指的是同一台手机**，并不是「必须用 127.0.0.1 才行」。连不上更常见是**浏览器限制**：部分浏览器（例如 Opera）对访问本机服务支持不好，换 Chrome 等通常能解决。

**你可以：**
1. 优先用 Chrome 打开 PWA；服务器地址用默认的 `http://localhost:8080` 即可（与 `http://127.0.0.1:8080` **等价**，不必特意改）。
2. 确认 bilirec-mobile 里后端已成功启动。

### 打开录制程序后还要填地址？

**你会看到：** 从 App 跳进网页，担心还要再填服务器。

**为什么：** App 负责启动手机里的后端；管理页是另一个程序。好消息是：**默认就是 `http://localhost:8080`**，一般**不用手改**，直接点登录即可（前提是后端已在 App 里启动成功）。

**你可以：** 打开 PWA 后先点登录；只有连不上时再检查地址或换浏览器（见上一条）。

## 性能与容量规划

### CPU 突然飙高怎么办？

**你会看到：** 任务管理器或手机通知栏里 CPU 百分比偶尔冲高。

**为什么：** 很多时候不是「算力不够」，而是**写盘太勤**——尤其 microSD、机械硬盘上，频繁 flush/sync 会造成 CPU 和 I/O 尖峰。订阅房间多、后台转码也会叠加一些占用。闲置时若大约每 10 秒有小尖峰，可能来自订阅开播轮询（subcheck）；可调 `SUBCHECK_CHECK_INTERVAL_SECS` / `SUBCHECK_SHARD_COUNT`（见 [录制配置](/zh-cn/configuration/recording/)）。

**你可以：**
1. 树莓派 + microSD：适当**加大写盘缓冲**、**拉长写盘间隔**、必要时关闭录制中整文件落盘（见 [I/O 调优](/zh-cn/configuration/io-tuning/) 与 [树莓派默认配置](/zh-cn/configuration/pi5-defaults/)）。
2. 订阅房间特别多时，把检查分组调多、一轮时间略拉长（见 [录制配置](/zh-cn/configuration/recording/)）。
3. 尽量不要边录边转码，或把转码挪到空闲时段。

### 录制时内存占用很高怎么办？

**你会看到：** 监控里内存在录播时明显上涨。

**为什么：** 直播数据会在内存里先排队再写盘，路数越多、画质越高（4K），占用越大。

**你可以：**
1. 按 [内存占用估算](/zh-cn/configuration/memory-estimation/) 调整缓冲相关配置（通常比直接减路数更合适）。
2. 不要只看 Docker 总内存——系统缓存文件也会让数字偏大。
3. 只有确认设备**确实达不到**公式或实测要求时，再考虑减少同时录制路数。

### 怎么知道我的设备能录几路？

**你会看到：** 不确定该设多少路、容器该分多少内存。

**为什么：** 上限取决于内存公式、CPU、以及磁盘（尤其是 microSD）三者里**最吃紧**的那一项。

**你可以：**
1. 用 [内存占用估算](/zh-cn/configuration/memory-estimation/) 算一下目标路数需要多少内存，容器内存留约两成余量。
2. 对照 [性能实测](/zh-cn/guides/performance-benchmark/) 里相近设备的数据。
3. 从 **1 路**开始试，逐步加到稳定为止。树莓派 + SD 卡还要参考 [树莓派默认配置](/zh-cn/configuration/pi5-defaults/)。

### 都停录、转码也结束了，内存为什么还不少？

**你会看到：** 任务都停了，Docker 或手机通知栏里内存仍比刚启动时高（例如五十～几百 MB）。

**为什么：**

1. **程序自己还留着一块「下次录制用」的缓冲**  
   停录后常见几十 MB，见 [性能实测「恢复闲置」](/zh-cn/guides/performance-benchmark/)（约 54 MB）。录过 4K 时，高画质相关部分大约 **15 分钟**不用后会逐步释放。

2. **刚录过很多路时，数字会暂时接近「录制高峰」**  
   例如 4～5 路同时录过，总量两三百 MB 可能和 [内存占用估算](/zh-cn/configuration/memory-estimation/) 里的峰值同一量级，**不一定是漏内存**。

3. **监控里的「总内存」常把「刚写过的录制文件缓存」也算进去**  
   这是系统为读写加速暂存的副本，**不是** bilirec 又多占了一份程序内存；内存紧张时系统会自己收回。详见 [内存占用估算 - 容器总内存虚高](/zh-cn/configuration/memory-estimation/#容器总内存虚高)。

**你可以：**

1. 对照 [性能实测](/zh-cn/guides/performance-benchmark/)「恢复闲置」— 在附近通常正常。
2. 想再低一点：略减缓冲相关配置（见 [内存占用估算](/zh-cn/configuration/memory-estimation/)）。
3. 若主要是监控「总内存」虚高：默认已开启录製中定期清理旧文件缓存；仍偏高时读 [内存占用估算](/zh-cn/configuration/memory-estimation/)。SSD/HDD 若更在意回放速度，可关闭（见下节）。

**何时要担心：** 停录很久仍**持续上涨**、或明显卡死 — 再查日志或提 issue；单次高峰后居高不下多数属于上面 1～3。

### 监控内存偏高？两个相关选项怎么理解？

**默认已开启**（一般不用改）：

| 环境变量 | 大白话 | 默认 |
| -------- | ------ | ---- |
| `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | **录製过程中**，每隔一段时间让系统忘掉「已经写好、暂时不会再读」的旧文件缓存 | `60`（秒） |
| `DROP_FILE_PAGE_CACHE` | **一段录完或转码结束后**，忘掉整份文件的系统缓存 | `true` |

两者配合，可避免 Docker / 手机监控里内存随录製一路顶到上限。代价是：刚录完立刻播放、或录製中回放已写好的前半段，可能要从磁盘重新读，略慢。

**冷释放不是「每 60 秒必清一次」：** 单段文件要长到约 96 MB 以上才会开始清理旧前缀；短 HLS 分段主要靠关档时的 `DROP_FILE_PAGE_CACHE`。详见 [I/O 调优 · 录制文件缓存释放](/zh-cn/configuration/io-tuning/#录制文件缓存释放linux--android)。

**适合保持默认：** 树莓派、1GB 容器、Android 后台、很少录完马上拖进度条。

**建议关闭（回放优先）：** SSD / NAS、录完马上要秒开播放。设 `DROP_FILE_PAGE_CACHE=false` 且 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=0`。

**和「周期性整文件落盘」(`LIVE_STREAM_WRITER_SYNC_PERIOD_SECS`) 别搞混：**

- 整文件落盘：为了**断电少丢数据**，录製中反复把整个文件刷进卡，**很伤 microSD**，也**压不住**监控内存。
- 上面的 60 秒选项：为了**监控好看**，只清理旧数据，**比整文件落盘省卡**；关档时仍会完整保存一次。
- 两个都设了：以整文件落盘为准，不再单独按 60 秒清缓存；**仅当 `COLD_CACHE_RELEASE_SECS>0`** 时，每次 fsync 成功后才顺便释放冷前缀。
- **落盘不会自动释放缓存**：`SYNC_PERIOD` 负责落盘；释放缓存靠 fsync 后的冷前缀释放、或独立的冷释放定时器、以及 `DROP_FILE_PAGE_CACHE`，分工不同。

若日志出现 `释放文件页缓存失败` 或 `冷缓存释放失败`，录制通常仍可继续，只是内存可能降不下来；见 [I/O 调优 · 录制文件缓存释放](/zh-cn/configuration/io-tuning/#录制文件缓存释放linux--android)。

详见 [环境变量说明](/zh-cn/configuration/env-reference/) · [I/O 调优](/zh-cn/configuration/io-tuning/)。

<Aside type="tip">
**简单说：** 停录后内存比刚开机高一些很常见；正在录的时候若紧张，优先调缓冲配置，别急着减录制路数。
</Aside>
