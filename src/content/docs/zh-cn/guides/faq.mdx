遇到报错时，先看界面或日志里的那句中文；若已经说清楚了，按提示操作即可。下面只整理**容易误会、或界面上看不出来原因**的情况。

## 后端录制

### 开了内网穿透和 HTTPS 后，程序启动就报错？

**你会看到：** 一启动就退出，日志里提到 FRP 或 HTTPS 配置不对。

**为什么：** 内网穿透连回本机的协议，需与你本机是否开 HTTPS 一致。两边开关对不上，程序会拒绝启动，避免连上却用不了。

**你可以：**
1. 若本机**没有**配证书，内网穿透用 HTTP 方式连回来（常见家用场景）。
2. 若本机**已经**配了证书走 HTTPS，内网穿透也要用 HTTPS 方式连回来。
3. 具体变量组合见 [远程访问（FRP）](/zh-cn/configuration/frp/)。

### FRP 服务器地址填错了？

**你会看到：** 日志里写「无效的 FRP_SERVER 格式」，或隧道一直连不上。

**为什么：** 地址一般宜写成 `主机名或IP:端口`，例如 `tunnel.bilirec.org:7000`。只写域名时，程序会默认用 7000 端口；带 `http://` 的网址可能无法正确解析。

**你可以：** 改成 `地址:端口` 后重启；自架 FRP 时还要填好 Token（见下一条）。

### 用自己的 FRP 服务器，外网还是进不来？

**你会看到：** 程序能启动，但日志里 FRP 初始化失败，或没有公网地址。

**为什么：** 连**自己的** FRP 时通常要填认证 Token；用官方公共服务时，只开 `FRP_ENABLED=true` 即可。

**你可以：**
1. 确认 `FRP_SERVER`、`FRP_BASE_DOMAIN` 与服务器一致。
2. 自架服务填上 `FRP_TOKEN`。
3. 详见 [远程访问（FRP）](/zh-cn/configuration/frp/)。

### 同一场直播里为什么会出现「标题-1」「标题-2」多个文件？

**你会看到：** 目录里除了 `标题-时间戳.flv`，还有 `标题-时间戳-1.flv`、`标题-时间戳-2.flv` 等同一场、**同一时间戳**下的分段。

**为什么：** 这是**直播轮转**（例如 PK、换画质导致流头变化）时程序自动切的新文件，属于正常设计。

**你可以：** 按文件名顺序观看即可。若你想了解轮转规则，见 [录制指南](/zh-cn/guides/recording/)。

### 断网重连后怎么又多了一个不同时间戳的文件？

**你会看到：** 网络卡了一下之后，目录里多了一个**新的时间戳**文件（文件名里没有 `-1`、`-2` 这种后缀）。

**为什么：** 断线后程序会重连并**新开一个文件**继续录，避免把两段不连续的流硬拼在一起。这和上一节说的「同时间戳带 -1/-2」不是一回事。

**你可以：** 按时间戳区分各段即可；若经常断网，先检查网络。重连相关配置见 [录制指南](/zh-cn/guides/recording/)。

### 开了「录完转 MP4」，却一直不转？

**你会看到：** 录制结束了，列表里仍是 flv/ts，或转码任务很久不动。

**为什么：** 默认为了省 CPU，**正在录别的房间时**不会跑本地转码；机器上也可能没装 ffmpeg，或没配云端转码。

**你可以：**
1. 等所有录制都停掉后再看是否开始转。
2. 确认已安装 ffmpeg，或配置了 CloudConvert。
3. 若要在录制时也转，需在配置里单独打开（见 [转换配置](/zh-cn/configuration/conversion/)）。

### 转码老是失败、反复重试？

**你会看到：** 转换任务一会儿失败一会儿又排队。

**为什么：** 常见原因有两类：一是转出来的 MP4 **太小**（不到 1MB 或不到原文件一半），程序会认为结果不可靠而重试；二是**源文件本身不完整或已损坏**（录制中断、磁盘问题等），怎么转都难成功。

**你可以：** 先确认源 flv/ts 能否正常播放；源文件有问题时只能保留原格式或重新录。详见 [转换配置](/zh-cn/configuration/conversion/)。

### 大文件走 CloudConvert 创建任务失败？

**你会看到：** 日志里和「公开地址」「预签名」有关的错误。

**为什么：** 云端要回来拉你的文件，需要填一个**能从外网访问到的完整网址**（`PUBLIC_BASE_URL`）；留空或只写内网地址时，CloudConvert 无法拉取文件。

**你可以：** 在配置里设好对外的 `http://` 或 `https://` 地址，或改用本机 ffmpeg 转小文件。见 [转换配置](/zh-cn/configuration/conversion/)。

## Web 管理界面

### 打开 app.bilirec.org，却连不上自己的录制服务？

**你会看到：** 页面能开，但登录或列表一直失败。

**为什么：** `app.bilirec.org` 是 **HTTPS 网页**，不是你的录制程序本身。网页要单独填后端地址才能连上。由于浏览器安全限制，从 HTTPS 页面去连 **HTTP 局域网地址**（如 `http://192.168.x.x:8080`）往往会被拦下；**HTTP 后端一般只能填本机** `http://localhost:8080` 或 `http://127.0.0.1:8080`。亦有可能是浏览器没有打开本地网络访问权限。

**你可以：**
1. 后端跑在本机时，用 `http://localhost:8080` 或 `http://127.0.0.1:8080`。
2. 检查浏览器是否打开了本地网络访问权限或者浏览器是否支持局域网访问。
3. 后端在局域网另一台机器上时，需要给后端配 **HTTPS**，或在同一台机器上用浏览器打开管理页；HTTPS 的 app.bilirec.org 无法直接连家里的 HTTP 地址。

### 用 app.bilirec.org 连远程后端，一直提示「会话已过期」？

**你会看到：** 账号密码明明正确，登录后刚进主界面几秒就弹出「会话已过期，请重新登录」；怎么重登都一样。后端本身可能一直正常。

**为什么：** [app.bilirec.org](https://app.bilirec.org/) 和你的后端地址（FRP 的 `https://xxx.tunnel.bilirec.org`、自架的 `https://api.example.com` 等）是**两个不同网址**。默认配置下，浏览器往往记不住登录状态，界面会先显示成功，几秒后又让你重登。若使用 FRP 且 bilirec 重启过，子域名可能已变，旧地址上的登录也会失效。

**根域名不同时**（例如自架前端 `https://bilirec.example.com` 连 FRP 的 `https://xxx.tunnel.bilirec.org`），浏览器还可能把登录状态当作第三方 Cookie 拦下；**app.bilirec.org 连 `*.tunnel.bilirec.org` 同属 `bilirec.org`，一般不会有这个问题**，优先检查 `PRODUCTION_MODE` 即可。

**你可以：**
1. 在后端设置 `PRODUCTION_MODE=true`，并配置 `USERNAME`、`PASSWORD`、固定的 `JWT_SECRET`；`FRONTEND_URL` 填你实际使用的管理页地址（官方页用 `https://app.bilirec.org`，自架前端填你的 HTTPS 域名），然后重启 bilirec。
2. 在登录页填入**当前**的后端地址（FRP 每次重启子域名可能变化；自架域名一般不变）。
3. **仅当前端与后端根域名不同**（自架前端 + FRP 等）且仍登录不上时：在浏览器里**允许第三方 Cookie**，或将前端与后端域名加入例外。Chrome / Edge：「设置 → 隐私和安全 → 第三方 Cookie」；Safari：关闭「阻止跨站跟踪」；Firefox：对前端域名放宽隐私保护。
4. 本机 `localhost` 开发不受此影响，无需改 `PRODUCTION_MODE`。

详见 [远程访问（FRP）— 通过 app.bilirec.org 登录](/zh-cn/guides/frp/#通过-appbilirecorg-登录)（自架 HTTPS 后端同样适用）与 [认证与安全](/zh-cn/configuration/auth-security/#何时需要生产模式)。

### 收不到开播推送通知？

**你会看到：** 浏览器或 PWA 里从没弹过通知。

**为什么：** 需要允许通知权限、成功连过后端，只读账号也不会推送。若用的是 **PWA 或 Chrome**，在内地网络下还可能因为**连不上 Google FCM** 而收不到推送。

**你可以：**
1. 在浏览器设置里允许本站通知。
2. 确认登录页的服务器地址正确并已保存。
3. 用管理员账号订阅推送；内地使用 PWA 或 Chrome 的用户若仍无推送，可改用 SSE 等方案。详见 [Web 界面](/zh-cn/guides/web-ui/) 与 [通知配置](/zh-cn/configuration/notifications/)。

## Android

### 录制保存到 SD 卡后变慢？

**你会看到：** 换到 SD 卡路径后，录制或界面变卡。

**为什么：** 外置 SD 通常比手机内置存储慢。若输出路径在 microSD 卡上，建议在 App **存储策略**里开启 **microSD 卡磨损保护**。

**你可以：** 在 App 的存储相关设置里按需开关；追求速度可改回内置存储。见 [Android 指南](/zh-cn/guides/android/) 与 [Android 默认配置](/zh-cn/configuration/android-defaults/)。

### 启动后切到后台，过一会儿服务就自己停了？

**你会看到：** 刚还能录，划走 App 或锁屏一会儿，通知栏服务没了或 Web 连不上。

**为什么：** 最常见是**没关电池优化**（或系统杀后台）。Android 对后台和省电管得很严，不按提示放行就容易被停掉。

**你可以：**
1. 按 App 提示**关闭电池优化**、允许通知（前台服务需要）。
2. 若选了外置存储路径，按提示授予存储权限。
3. 建议避免让系统「清理后台」。见 [Android 指南](/zh-cn/guides/android/)。

### 用 bilirec-mobile 打开 PWA 后，连不上本机后端？

**你会看到：** 默认地址连不上，或换浏览器才好使。

**为什么：** 连不上更常见是**浏览器限制**：部分浏览器（例如 Opera）对访问本机服务支持不好，换 Chrome 等通常能解决。

**你可以：**
1. 优先用 Chrome 打开 PWA；服务器地址用默认的 `http://localhost:8080` 或 `http://127.0.0.1:8080` 即可。
2. 确认 bilirec-mobile 里后端已成功启动。

### 打开录制程序后还要填地址？

**你会看到：** 从 App 跳进网页，担心还要再填服务器。

**为什么：** App 负责启动手机里的后端；管理页是另一个程序。正常来说登录界面可以直接点击登入（默认就是本地地址），之后再也不需要填地址了。

**你可以：** 进入登录界面后直接点击登录，或安装为PWA之后直接打开PWA启动。

## 性能与容量规划

### CPU 突然飙高怎么办？

**你会看到：** 任务管理器或手机通知栏里 CPU 百分比偶尔冲高。

先对照下表判断尖峰来源，再调对应环境变量：

| 可能来源 | 你怎么分辨 | 相关环境变量 | 建议方向 |
| -------- | ---------- | ------------ | -------- |
| **写盘 / flush 尖峰** | 正在录制时周期性冲高；microSD、机械盘更明显 | `LIVE_STREAM_WRITER_BUFFER_SIZE`<br />`LIVE_STREAM_WRITER_FLUSH_PERIOD_SECS`<br />`LIVE_STREAM_WRITER_SYNC_PERIOD_SECS` | 适当加大合并缓冲、拉长 flush 间隔；**保持** `SYNC_PERIOD=0`（开周期性整文件落盘会更吃 CPU） |
| **多路抢同一张盘** | 多路同时录到同一张 SD / HDD 时尖峰叠加 | `SEQUENTIAL_WRITE` | 同一张卡上多路录制时保持 `true`（Android 输出到 microSD 时在 App 开启 **microSD 卡磨损保护**） |
| **订阅开播轮询** | **闲置**时大约每隔固定秒数有小尖峰 | `SUBCHECK_ROOMS_PER_SHARD`<br />`SUBCHECK_MIN_INTERVAL_SECS` | 默认已按订阅规模自动调度；仍偏高可略增大每组房间数或周期下限 |
| **后台转码** | 转码任务进行中 CPU 持续偏高 | `FFMPEG_MAX_CONCURRENT_TASKS`<br />`FFMPEG_ALLOW_DURING_RECORDING`<br />`FFMPEG_CHECK_INTERVAL_SECS` | 建议避免边录边转；并发转码可保持 `1`；详见 [转换配置](/zh-cn/configuration/conversion/) |

**你可以：**
1. 树莓派 + microSD：从 [I/O 调优](/zh-cn/configuration/io-tuning/) 与 [树莓派默认配置](/zh-cn/configuration/pi5-defaults/) 调整写盘相关项。
2. 订阅房间特别多：见 [录制配置 · 订阅房间状态检查](/zh-cn/configuration/recording/#订阅房间状态检查)（默认自动调度，一般无需随订阅改环境变量）。
3. 转码与录制错峰：见 [转换配置](/zh-cn/configuration/conversion/)。

### HDD NAS 用 ZFS，CPU 比平时高一点？

**你会看到：** 同样路数，输出目录在机械盘 **ZFS 池** 上时，CPU 比 EXT4/XFS 等同场景多几个百分点，但录制仍顺畅。

**为什么：** ZFS 要做校验、元数据与写入合并，多路录制时这部分开销会叠加。这是 **ZFS 本身的代价**，与是否设 `SEQUENTIAL_WRITE=false` **无直接关系**。

**你可以：**
1. 一般不必处理，属正常现象。
2. 若是**周期性尖峰**或 NAS 明显变卡，再对照上一节 [CPU 突然飙高](/zh-cn/guides/faq/#cpu-突然飙高怎么办) 排查写盘、多路抢盘等项。
3. ZFS 池调参见 [HDD/NAS 调优](/zh-cn/configuration/hdd-nas-tuning/)。

### 录制时内存占用很高怎么办？

**你会看到：** 监控里内存在录播时缓缓上涨。

**为什么：** 常见是 **程序本身的缓冲**（路数越多、4K 越高越大）或 **系统页缓存**（文件被缓存到内存里）。先对照 [内存占用估算 · 监控里的内存从哪来](/zh-cn/configuration/memory-estimation/#监控里的内存从哪来) 判断来源。

**你可以：**
1. 若是程序本身的缓冲：按 [内存占用估算](/zh-cn/configuration/memory-estimation/) 调整 `READ_STREAM_*`、`LIVE_STREAM_WRITER_*`（通常比直接减路数更合适）。
2. 若是系统页缓存：默认已开冷释放与关档释放；仍偏高见 [I/O 调优](/zh-cn/configuration/io-tuning/#录制文件缓存释放linux--android) 或 [常见问题 · 监控内存偏高](/zh-cn/guides/faq/#监控内存偏高两个相关选项怎么理解)。
3. 只有确认设备**确实达不到**公式或实测要求时，再考虑减少 `MAX_CONCURRENT_RECORDINGS`。

### 怎么知道我的设备能录几路？

**你会看到：** 不确定该设多少路、容器该分多少内存。

**为什么：** 上限取决于内存公式、CPU、以及磁盘（尤其是 microSD）三者里**最吃紧**的那一项。

**你可以：**
1. 用 [内存占用估算](/zh-cn/configuration/memory-estimation/) 算一下目标路数需要多少内存，容器内存留约两成余量。
2. 对照 [性能实测](/zh-cn/guides/performance-benchmark/) 里相近设备的数据。
3. 从 **1 路**开始试，逐步加到稳定为止。树莓派 + SD 卡还要参考 [树莓派默认配置](/zh-cn/configuration/pi5-defaults/)。

### 都停录、转码也结束了，内存为什么还不少？

**你会看到：** 任务都停了，Docker 或手机通知栏里内存仍比刚启动时高（例如五十～几百 MB）。

**为什么：**

1. **内存池预热** — bilirec 会把读入、写盘用的内存块留在进程里复用，减轻下次录制时的分配开销。停录后常见几十 MB 的「底座」，见 [性能实测「恢复闲置」](/zh-cn/guides/performance-benchmark/)（约 54 MB）。录过 4K / 杜比时，高画质那套池子大约 **15 分钟**不用后会自动释放。
2. **系统页缓存** — 数据写入磁盘后，Linux / Android 往往还会在内存里留一份文件副本（页缓存），用来加速以后的读写。Docker 或手机监控的「总内存」会把这份算进去。

**你可以调：**

| 想改善什么 | 环境变量 | 方向 |
| ---------- | -------- | ---- |
| 让系统忘掉文件页缓存 | `DROP_FILE_PAGE_CACHE` | 保持 `true`（默认已开）；关档 / 转码后整份释放 |
| 录制中更勤地清旧页缓存 | `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | 改小间隔（如 `60` → `30`）；`0` = 关闭录制中冷释放 |
| 降低内存池预热占用 | `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE`<br />`READ_STREAM_CHAN_BUFFER_SIZE`<br />`LIVE_STREAM_WRITER_BUFFER_SIZE` 等 | 略减小缓冲，见 [内存占用估算](/zh-cn/configuration/memory-estimation/#想省内存优先改哪几个) |

冷释放细节（何时才真正执行、与整文件落盘的区别）见 [I/O 调优 · 录制文件缓存释放](/zh-cn/configuration/io-tuning/#录制文件缓存释放linux--android)。

**何时要担心：** 停录很久仍**持续上涨**或明显卡死 — 再查日志；多数情况下属于上面三类正常残留。

### 在线播放或下载后，监控内存一直很高？

浏览器在线播放、或下载已录好的 MP4 时，系统会为读盘加速在内存里**再留一份文件副本**（与录制写入类似，还是属于系统页缓存）。播得越久、拖进度条越频繁，这份缓存可能越大；**停播后**若监控仍居高不下，多半是缓存还没被系统收回。

**默认行为（`DROP_FILE_PAGE_CACHE=true`）：** 最后一次播放或下载请求结束后，会等待 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` 秒（默认 **60**）再释放该文件的缓存；这段时间内若再次播放同一文件，计时会重新算起。若你把该变量设为 `0`（仅关闭**录制中**冷释放），播放/下载空闲仍按默认 **60** 秒等待后释放，避免与关档释放（`DROP_FILE_PAGE_CACHE`）绑死。

**局限：** 长时间连续播放、Range 请求不断到来时，计时会被反复重置，**播放过程中**监控仍可能上涨；本机制主要改善**停播 / 下载完成后**曲线不降的问题。

**若更在意播放流畅、不在意监控数字：** 设 `DROP_FILE_PAGE_CACHE=false`（同时也会影响关档释放，见下节）。

### 监控内存偏高？两个相关选项怎么理解？

**默认已开启**（一般不用改）：

| 环境变量 | 说明 | 默认 |
| -------- | ------ | ---- |
| `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | **录制过程中**，每隔一段时间让系统释放「已经写好、暂时不会再读」的旧文件缓存 | `60`（秒） |
| `DROP_FILE_PAGE_CACHE` | **一段录完或转码结束后**，释放整份文件的系统页缓存 | `true` |

两者配合，可避免 Docker / 手机监控里内存随录制一路顶到上限。代价是：刚录完立刻播放、或录制中回放已写好的前半段，可能要从磁盘重新读，略慢。

**冷释放不是「每 60 秒必清一次」：** 单段文件要长到约 96 MB 以上才会开始清理旧前缀；短 HLS 分段主要靠关档时的 `DROP_FILE_PAGE_CACHE`。详见 [I/O 调优 · 录制文件缓存释放](/zh-cn/configuration/io-tuning/#录制文件缓存释放linux--android)。

**适合保持默认：** 树莓派、1GB 容器、Android 后台、很少录完马上拖进度条。

**建议关闭（回放优先）：** SSD / NAS、录完马上要秒开播放。设 `DROP_FILE_PAGE_CACHE=false` 且 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=0`。

**和「周期性整文件落盘」(`LIVE_STREAM_WRITER_SYNC_PERIOD_SECS`) 的区别：**

- `LIVE_STREAM_WRITER_SYNC_PERIOD_SECS` 只是为了**断电少丢数据**，录制中反复把整个文件刷进卡，**很伤 microSD**，也**压不住**监控内存。
- 两个都设置了：以整文件落盘为准，不再单独按 60 秒清缓存；**仅当 `COLD_CACHE_RELEASE_SECS>0`** 时，每次落盘成功后才顺便释放当前页缓存。
- `LIVE_STREAM_WRITER_SYNC_PERIOD_SECS` **本身不会自动释放缓存**：它只负责落盘；释放缓存靠落盘后的页缓存释放、需要配合设置 `DROP_FILE_PAGE_CACHE=true` 才能生效。

若日志出现 `释放文件页缓存失败` 或 `冷缓存释放失败`，录制通常仍可继续，只是内存可能降不下来；见 [I/O 调优 · 录制文件缓存释放](/zh-cn/configuration/io-tuning/#录制文件缓存释放linux--android)。
