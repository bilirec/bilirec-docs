录制时，直播数据会在内存里「排队」：先从网络读进来，再排队等待写入磁盘。用下面公式估算**给容器留多少内存**即可（偏保守的上限；实际监控往往略低，见 [性能实测](/zh-cn/guides/performance-benchmark/)）。

## 会影响内存占用的环境变量

每路录制时，内存里同时等着处理的数据大致分三块：**网络读入排队**、**等待写盘排队**、**写盘前合并缓冲**。录 **1080p** 用不带 `_HIGH` 的变量；录 **4K / 杜比**（`qn ≥ 20000`）会自动改用带 `_HIGH` 的高画质变量。

| 作用 | 1080p 环境变量 | 4K / 杜比环境变量 | 默认值 |
| ---- | -------------- | ----------------- | ------ |
| 读入时能排多少「块」 | `READ_STREAM_CHAN_BUFFER_SIZE` | `READ_STREAM_CHAN_BUFFER_SIZE_HIGH` | 16 / 48 |
| 读入时每块多大 | `READ_STREAM_BYTES_POOL_SIZE` | `READ_STREAM_BYTES_POOL_SIZE_HIGH` | 512 KB / 1 MB |
| 写盘前能排多少「块」 | `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE` | （两档共用） | 64 |
| 写盘前每块多大 | `LIVE_STREAM_WRITER_BYTES_POOL_SIZE` | `LIVE_STREAM_WRITER_BYTES_POOL_SIZE_HIGH` | 512 KB / 1 MB |
| 合并后一次写多少 | `LIVE_STREAM_WRITER_BUFFER_SIZE` | （两档共用） | 8 MB |

路数由 `MAX_CONCURRENT_RECORDINGS` 决定，直接代入下方公式里的 N。

<Aside type="note">
读入和写盘各有一份数据在排队，下面公式已把两侧都算上。
</Aside>

## 怎么算（HTTP-FLV，最常见）

```
总峰值 ≈ 43~50 MB + N × 单路排队峰值

单路排队峰值 = 读入排队 + 写盘排队 + 合并缓冲

读入排队 = (READ_STREAM_CHAN_BUFFER_SIZE + 1) × READ_STREAM_BYTES_POOL_SIZE
写盘排队 = (LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE + 1) × LIVE_STREAM_WRITER_BYTES_POOL_SIZE
合并缓冲 = LIVE_STREAM_WRITER_BUFFER_SIZE
```

其中 **43~50 MB** 是程序本身和系统固定开销（和缓冲配置几乎无关，也**不能通过环境变量调小**）。录 4K / 杜比时，读侧两个变量换成 `*_HIGH`，写侧块大小换成 `LIVE_STREAM_WRITER_BYTES_POOL_SIZE_HIGH`。

### 默认 1080p 示例

| 路数 N | 计算 | 预计总峰值 |
| ------ | ---- | ---------- |
| 1 | `(43~50) + 1×49` | 约 92~99 MB |
| 3 | `(43~50) + 3×49` | 约 190~197 MB |
| 5 | `(43~50) + 5×49` | 约 288~295 MB |

（单路排队：读入约 8.5 MB + 写盘约 32.5 MB + 合并 8 MB ≈ 49 MB）

### 默认 4K 单路

| 路数 N | 计算 | 预计总峰值 |
| ------ | ---- | ---------- |
| 1 | `(43~50) + 1×122` | 约 165~172 MB |

（单路排队：读入约 49 MB + 写盘约 65 MB + 合并 8 MB ≈ 122 MB）

4K / 杜比使用 `*_HIGH` 读侧变量与 `LIVE_STREAM_WRITER_BYTES_POOL_SIZE_HIGH` 时的默认估算。

## HLS 怎么算

HLS 读进来的是**一整段分片**（大小由 m3u8 决定，常见 2–6 MB），读侧用分片体积 `S` 估算：

```
总峰值 ≈ 43~50 MB + N × (读入排队 + 写盘排队 + 合并缓冲)

读入排队 ≈ (READ_STREAM_CHAN_BUFFER_SIZE + 3) × S
```

写侧仍按上一节的写盘排队 + 合并缓冲计算。录 4K 时读入排队深度使用 `READ_STREAM_CHAN_BUFFER_SIZE_HIGH`。

## 其他说明

- **容器总内存虚高**：写入磁盘后，系统可能把录制文件再缓存一份以加速读写，**容器监控的总用量**因此会远高于 bilirec 实际占用。默认已开启 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=60`（录制中清理旧文件缓存）与 `DROP_FILE_PAGE_CACHE=true`（关档后释放整份文件缓存），**监控曲线**通常更接近 [性能实测](/zh-cn/guides/performance-benchmark/)（1 路约 65 MB、5 路约 165 MB）。若关闭上述选项，长录仍可能虚高；规划 **limit** 仍以上表公式为准。
- **停录后仍偏高**：程序会保留一部分缓冲供下次录制复用（见[性能实测](/zh-cn/guides/performance-benchmark/)「恢复闲置 ~54 MB」），属正常。

## 想省内存，优先改哪几个

按效果从大到小；改小后**你会注意到**：同时能扛住的网络波动变短，但内存占用下降。

1. `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE`（默认 64 → 32，写盘排队约减半）
2. `READ_STREAM_CHAN_BUFFER_SIZE`（仅 FLV；HLS 同理改读入排队深度或 `_HIGH` 版）
3. `LIVE_STREAM_WRITER_BUFFER_SIZE`（合并缓冲变小，写盘更频繁，SD 卡磨损可能增加）
4. `MAX_CONCURRENT_RECORDINGS`（直接减少同时录制的路数）

<Aside type="tip">
**常见问题：** 录制时内存高、停录后仍偏高、能录几路——见 [常见问题](/zh-cn/guides/faq/#性能与容量规划)。
</Aside>
