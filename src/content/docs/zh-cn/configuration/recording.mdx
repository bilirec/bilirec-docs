## 并发与时长

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `MAX_CONCURRENT_RECORDINGS` | 最大同时录制数 | `3` |
| `MAX_RECORDING_HOURS` | 单次录制最长时间（小时） | `5` |
| `MAX_RECOVERY_ATTEMPTS` | 单次录制最大重连尝试次数（流仍在线但连接失败时） | `15` |
| `MAX_RETRY_MINUTES` | 直播下线后判断容忍时间（分钟）；下线期间持续重试，超时后结束录制 | `10` |
| `RECORDING_RECOVERY_DURATION` | 断流重试**成功并恢复录制**后，录制时长如何计算：`preserve`（沿用开始录制时的总时长，不延长）或 `reset`（从重试成功时重新计算） | `preserve` |

`RECORDING_RECOVERY_DURATION` 只影响流仍在线、重连恢复成功的情况，不改变 `MAX_RETRY_MINUTES` 的下线重试逻辑。若你设置了单次录制时长上限（API 的 `duration_minutes` 或 `MAX_RECORDING_HOURS`），`preserve` 表示录制时长从**第一次开始录制**起算、不延长；`reset` 表示每次重连成功后重新享有完整录制时长，适合担心管理员强切新场次时漏录的场景。

断流恢复后会**用新的时间戳开一个新文件**（例如 `标题-20260711_160512.flv`），不会覆盖断流前的录制，也不会和直播轮转产生的 `-1`/`-2` 后缀文件混用同一套命名。`preserve` 与 `reset` 只影响「还能录多久」，不影响文件名规则。

<Aside type="tip">
**文件名与标题：**
- 开录时的房间标题会写入文件名；录制中途改标题不会改写已在写的文件名。
- 断流恢复后的新文件会用恢复当时的房间标题。
</Aside>

`MAX_CONCURRENT_RECORDINGS` 直接影响内存峰值。树莓派默认 `3`（1080p 默认缓冲下可验证至 5 路），SSD 环境可放宽至 `10`。规划容器上限可参考 [内存占用估算](/zh-cn/configuration/memory-estimation/)，详见 [树莓派默认](/zh-cn/configuration/pi5-defaults/) 与 [SSD 调优](/zh-cn/configuration/ssd-tuning/)。

调高同时录制路数时，若保持 `SEQUENTIAL_WRITE=true`（microSD、机械盘、NAS 的默认建议），建议按路数比例同步增大 `LIVE_STREAM_WRITER_FLUSH_PERIOD_SECS`，以免每路写盘间隔被压得过短、写盘更勤或 CPU 尖峰更明显。换算方式与示例见 [I/O 调优 · 多路录制与 flush 周期](/zh-cn/configuration/io-tuning/#多路录制与-flush-周期)。Android 开启 microSD 卡磨损保护时会自动处理，见 [Android 默认配置](/zh-cn/configuration/android-defaults/)。

## 存储路径

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `OUTPUT_DIR` | 录制文件保存目录 | `records` |
| `SECRET_DIR` | Cookie、Token、VAPID 密钥目录 | `secrets` |
| `DATABASE_DIR` | bbolt 数据库目录 | `database` |

Docker 部署时建议挂载为卷：

```bash
-v /path/to/records:/app/records
-v /path/to/secrets:/app/secrets
-v /path/to/database:/app/database
```

## 磁盘空间

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `MIN_DISK_SPACE_BYTES` | 录制所需最小磁盘空间（字节） | `5368709120` (5 GB) |

低于此值将拒绝新录制任务。Android 默认使用更低的阈值。

## 订阅房间状态检查

为任一订阅房间开启 **开播通知**（`notify`）或 **自动录制**（`auto_record`）后，后台会定期检查直播状态。检测到新一场直播时，按房间配置推送通知和/或启动录制。服务启动时先全量检查一次，之后将相关房间分成若干组，在完整周期内分组轮流检查，降低 CPU 峰值。

程序会按当前需检查的房间数（开启通知或自动录制的订阅）自动计算分组数与完整轮询周期，订阅增删后**无需改环境变量或重启**。每组内房间通过批量 API 一次拉取；默认约每 50 间一组、每组间隔约 10 秒触发。≤50 间时通常 60 秒一轮；订阅更多时分组与周期会相应增大。

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `SUBCHECK_ROOMS_PER_SHARD` | 每组目标房间数 | `50` |
| `SUBCHECK_TICK_SECS` | 目标分组触发间隔（秒） | `10` |
| `SUBCHECK_MIN_INTERVAL_SECS` | 完整轮询周期下限（秒） | `60` |
| `SUBCHECK_MAX_INTERVAL_SECS` | 完整轮询周期上限（秒） | `300` |
| `SUBCHECK_MAX_SHARDS` | 分组数上限 | `32` |
| `SUBCHECK_JITTER_SECS` | 每组轮询触发前的最大随机延迟（秒），打散固定节奏、降低被 B 站反爬识别风险 | `2` |

订阅极多且仍感到 CPU 尖峰偏高时，可略增大 `SUBCHECK_ROOMS_PER_SHARD`（每组更多房间、分组更少）或略增大 `SUBCHECK_MIN_INTERVAL_SECS`（接受稍慢的开播检测）。若频繁触发风控或日志出现切换账号凭证模式，可适当增大 `SUBCHECK_JITTER_SECS`。使用流程见 [录制直播间](/zh-cn/guides/recording/#订阅房间)。

## 直播间信息查询（风控自动切换）

查询房间是否在播、拉取房间标题封面等信息时（包括订阅轮询、Web 界面手动刷新、开始录制前的状态检查），程序**默认先用匿名方式**向 B 站请求，不携带你的登录 Cookie，以降低账号被风控的风险。

当 B 站返回表示「当前网络或请求被限制」的 API 错误码时（例如常见的 `-352` 等，HTTP 仍可能是 200），程序会**自动改用已登录账号的 Cookie 重试同一次查询**，并在日志中提示：

- **警告**：首次触发切换时，例如「直播间信息请求触发风控，已切换为账号凭证模式」
- **信息**：连续多次触发后锁定为账号模式，或探测恢复匿名模式时的提示

你**无需配置**也**不会在 Web 界面看到横幅**；若关心状态，查看后端日志即可。

| 说明 | 内容 |
|------|------|
| 适用请求 | 所有「直播间基础信息」类查询（与订阅轮询共用同一机制） |
| 需已登录 | `BILIBILI_LOGIN_MODE=controller` 或 `startup`，且已完成 B 站扫码登录；否则风控时无法自动切换，只能等网络恢复 |
| `anonymous` 模式 | 全程匿名，无 Cookie 可切换；被风控时查询可能失败直至 IP 限制解除 |
| 账号安全 | 仅在匿名请求被判定为风控时才使用 Cookie；正常网络下订阅轮询不消耗账号配额 |
| 恢复 | 进入账号模式后，约每 5 分钟用匿名方式探测一次；连续探测成功后会自动回到匿名模式 |

明确属于「房间不存在」、参数错误（`400` / `404`）或服务端错误（`500` 及以上）的响应**不会**触发 Cookie 重试。

</Aside>
