錄製時，直播數據會在記憶體裡「排隊」：先從網路讀進來，再排隊等待寫入磁碟。用下面公式估算**給容器留多少記憶體**即可（偏保守的上限；實際監控往往略低，見 [性能實測](/zh-tw/guides/performance-benchmark/)）。

## 會影響記憶體佔用的環境變數

每路錄製時，記憶體裡同時等著處理的數據大致分三塊：**網路讀入排隊**、**等待寫盤排隊**、**寫盤前合併緩衝**。錄 **1080p** 用不帶 `_HIGH` 的變數；錄 **4K / 杜比**（`qn ≥ 20000`）會自動改用帶 `_HIGH` 的高畫質變數。

| 作用 | 1080p 環境變數 | 4K / 杜比環境變數 | 預設值 |
| ---- | -------------- | ----------------- | ------ |
| 讀入時能排多少「塊」 | `READ_STREAM_CHAN_BUFFER_SIZE` | `READ_STREAM_CHAN_BUFFER_SIZE_HIGH` | 16 / 48 |
| 讀入時每塊多大 | `READ_STREAM_BYTES_POOL_SIZE` | `READ_STREAM_BYTES_POOL_SIZE_HIGH` | 512 KB / 1 MB |
| 寫盤前能排多少「塊」 | `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE` | （兩檔共用） | 64 |
| 寫盤前每塊多大 | `LIVE_STREAM_WRITER_BYTES_POOL_SIZE` | `LIVE_STREAM_WRITER_BYTES_POOL_SIZE_HIGH` | 512 KB / 1 MB |
| 合併後一次寫多少 | `LIVE_STREAM_WRITER_BUFFER_SIZE` | （兩檔共用） | 8 MB |

路數由 `MAX_CONCURRENT_RECORDINGS` 決定，直接代入下方公式裡的 N。

<Aside type="note">
讀入和寫盤各有一份數據在排隊，下面公式已把兩側都算上。
</Aside>

## 怎麼算（HTTP-FLV，最常見）

```
總峰值 ≈ 43~50 MB + N × 單路排隊峰值

單路排隊峰值 = 讀入排隊 + 寫盤排隊 + 合併緩衝

讀入排隊 = (READ_STREAM_CHAN_BUFFER_SIZE + 1) × READ_STREAM_BYTES_POOL_SIZE
寫盤排隊 = (LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE + 1) × LIVE_STREAM_WRITER_BYTES_POOL_SIZE
合併緩衝 = LIVE_STREAM_WRITER_BUFFER_SIZE
```

其中 **43~50 MB** 是程式本身和系統固定開銷（和緩衝設定幾乎無關，也**不能通過環境變數調小**）。錄 4K / 杜比時，讀側兩個變數換成 `*_HIGH`，寫側塊大小換成 `LIVE_STREAM_WRITER_BYTES_POOL_SIZE_HIGH`。

### 預設 1080p 示例

| 路數 N | 計算 | 預計總峰值 |
| ------ | ---- | ---------- |
| 1 | `(43~50) + 1×49` | 約 92~99 MB |
| 3 | `(43~50) + 3×49` | 約 190~197 MB |
| 5 | `(43~50) + 5×49` | 約 288~295 MB |

（單路排隊：讀入約 8.5 MB + 寫盤約 32.5 MB + 合併 8 MB ≈ 49 MB）

### 預設 4K 單路

| 路數 N | 計算 | 預計總峰值 |
| ------ | ---- | ---------- |
| 1 | `(43~50) + 1×122` | 約 165~172 MB |

（單路排隊：讀入約 49 MB + 寫盤約 65 MB + 合併 8 MB ≈ 122 MB）

4K / 杜比使用 `*_HIGH` 讀側變數與 `LIVE_STREAM_WRITER_BYTES_POOL_SIZE_HIGH` 時的預設估算。

## HLS 怎麼算

HLS 讀進來的是**一整段分片**（大小由 m3u8 決定，常見 2–6 MB），讀側用分片體積 `S` 估算：

```
總峰值 ≈ 43~50 MB + N × (讀入排隊 + 寫盤排隊 + 合併緩衝)

讀入排隊 ≈ (READ_STREAM_CHAN_BUFFER_SIZE + 3) × S
```

寫側仍按上一節的寫盤排隊 + 合併緩衝計算。錄 4K 時讀入排隊深度使用 `READ_STREAM_CHAN_BUFFER_SIZE_HIGH`。

## 其他說明

### 監控裡的記憶體從哪來？

容器或手機通知欄顯示的「總記憶體」，往往不只包含 bilirec 程式自己的緩衝。先判斷主要來源，再對症調整：

| 可能來源 | 你怎麼分辨 | 相關環境變數 | 建議方向 |
| -------- | ---------- | ------------ | -------- |
| **程式緩衝（讀入 / 寫盤排隊）** | 錄播時隨路數、畫質上漲；停錄後若仍偏高但穩定，可能還留著複用緩衝 | `READ_STREAM_*`、`LIVE_STREAM_WRITER_*`、`MAX_CONCURRENT_RECORDINGS` | 按上文公式估算；優先減小 `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE`、`READ_STREAM_CHAN_BUFFER_SIZE`，見下文「想省記憶體」 |
| **系統頁緩存（錄製寫入）** | 長錄、多路時監控遠高於 [性能實測](/zh-tw/guides/performance-benchmark/)，關檔後仍虛高 | `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS`、`DROP_FILE_PAGE_CACHE` | 預設已開啟；仍偏高見 [I/O 調優 · 錄製檔案緩存釋放](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android) |
| **系統頁緩存（在線播放 / 下載）** | 播完或下完仍居高不下 | 同上；播放空閒等待亦用 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | 見 [常見問題 · 在線播放或下載後](/zh-tw/guides/faq/#在線播放或下載後監控記憶體一直很高) |
| **停錄後緩衝複用** | 任務都停了，仍比剛啟動高約幾十 MB，且長期穩定 | （屬正常行為，無單獨開關） | 對照性能實測「恢復閒置」；想略降可調小緩衝相關變數 |

規劃 **Docker 記憶體 limit** 建議用上文公式（偏保守）；對照監控曲線可參考性能實測。

- **容器總記憶體虛高**：寫入磁碟後，系統可能把錄製檔案再緩存一份以加速讀寫，**容器監控的總用量**因此會遠高於 bilirec 實際佔用。預設已開啟 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=60`（錄製中清理舊檔案緩存）與 `DROP_FILE_PAGE_CACHE=true`（關檔後釋放整份檔案緩存），**監控曲線**通常更接近 [性能實測](/zh-tw/guides/performance-benchmark/)；規劃 **limit** 仍以上表公式為準。
- **停錄後仍偏高**：程式會保留一部分緩衝供下次錄製複用（見[性能實測](/zh-tw/guides/performance-benchmark/)「恢復閒置 ~54 MB」），屬正常。

## 想省記憶體，優先改哪幾個

按效果從大到小；改小後**你會注意到**：同時能扛住的網路波動變短，但記憶體佔用下降。

1. `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE`（預設 64 → 32，寫盤排隊約減半）
2. `READ_STREAM_CHAN_BUFFER_SIZE`（僅 FLV；HLS 同理改讀入排隊深度或 `_HIGH` 版）
3. `LIVE_STREAM_WRITER_BUFFER_SIZE`（合併緩衝變小，寫盤更頻繁，SD 卡磨損可能增加）
4. `MAX_CONCURRENT_RECORDINGS`（直接減少同時錄製的路數）

</Aside>
