## 並發與時長

| 變數 | 說明 | 預設值 |
|------|------|--------|
| `MAX_CONCURRENT_RECORDINGS` | 最大同時錄製數 | `3` |
| `MAX_RECORDING_HOURS` | 單次錄製最長時間（小時） | `5` |
| `MAX_RECOVERY_ATTEMPTS` | 單次錄製最大重連嘗試次數（流仍在線但連接失敗時） | `15` |
| `MAX_RETRY_MINUTES` | 直播下線後判斷容忍時間（分鐘）；下線期間持續重試，超時後結束錄製 | `10` |
| `RECORDING_RECOVERY_DURATION` | 斷流重試**成功並恢復錄製**後，錄製時長如何計算：`preserve`（沿用開始錄製時的總時長，不延長）或 `reset`（從重試成功時重新計算） | `preserve` |

`RECORDING_RECOVERY_DURATION` 只影響流仍在線、重連恢復成功的情況，不改變 `MAX_RETRY_MINUTES` 的下線重試邏輯。若你設定了單次錄製時長上限（API 的 `duration_minutes` 或 `MAX_RECORDING_HOURS`），`preserve` 表示錄製時長從**第一次開始錄製**起算、不延長；`reset` 表示每次重連成功後重新享有完整錄製時長，適合擔心管理員強切新場次時漏錄的場景。

斷流恢復後會**用新的時間戳開一個新檔案**（例如 `標題-20260711_160512.flv`），不會覆蓋斷流前的錄製，也不會和直播輪轉產生的 `-1`/`-2` 後綴檔案混用同一套命名。`preserve` 與 `reset` 只影響「還能錄多久」，不影響檔案名規則。

`MAX_CONCURRENT_RECORDINGS` 直接影響記憶體峰值。樹莓派預設 `3`（1080p 預設緩衝下可驗證至 5 路），SSD 環境可放寬至 `10`。規劃容器上限可參考 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/)，詳見 [樹莓派預設](/zh-tw/configuration/pi5-defaults/) 與 [SSD 調優](/zh-tw/configuration/ssd-tuning/)。

調高同時錄製路數時，若保持 `SEQUENTIAL_WRITE=true`（microSD、機械盤、NAS 的預設建議），建議按路數比例同步增大 `LIVE_STREAM_WRITER_FLUSH_PERIOD_SECS`，以免每路寫盤間隔被壓得過短、寫盤更勤或 CPU 尖峰更明顯。換算方式與示例見 [I/O 調優 · 多路錄製與 flush 週期](/zh-tw/configuration/io-tuning/#多路錄製與-flush-週期)。Android 開啟 microSD 卡磨損保護時會自動處理，見 [Android 預設設定](/zh-tw/configuration/android-defaults/)。

## 儲存路徑

| 變數 | 說明 | 預設值 |
|------|------|--------|
| `OUTPUT_DIR` | 錄製檔案保存目錄 | `records` |
| `SECRET_DIR` | Cookie、Token、VAPID 密鑰目錄 | `secrets` |
| `DATABASE_DIR` | bbolt 數據庫目錄 | `database` |

Docker 部署時建議掛載為卷：

```bash
-v /path/to/records:/app/records
-v /path/to/secrets:/app/secrets
-v /path/to/database:/app/database
```

## 磁碟空間

| 變數 | 說明 | 預設值 |
|------|------|--------|
| `MIN_DISK_SPACE_BYTES` | 錄製所需最小磁碟空間（字節） | `5368709120` (5 GB) |

低於此值將拒絕新錄製任務。Android 預設使用更低的閾值。

## 訂閱房間狀態檢查

為任一訂閱房間開啟 **開播通知**（`notify`）或 **自動錄製**（`auto_record`）後，後台會定期檢查直播狀態。檢測到新一場直播時，按房間設定推送通知和/或啟動錄製。服務啟動時先全量檢查一次，之後將相關房間分成若干組，在完整週期內分組輪流檢查，降低 CPU 峰值。

程式會按當前需檢查的房間數（開啟通知或自動錄製的訂閱）自動計算分組數與完整輪詢週期，訂閱增刪後**無需改環境變數或重啟**。每組內房間通過批量 API 一次拉取；預設約每 50 間一組、每組間隔約 10 秒觸發。≤50 間時通常 60 秒一輪；訂閱更多時分組與週期會相應增大。

| 變數 | 說明 | 預設值 |
|------|------|--------|
| `SUBCHECK_ROOMS_PER_SHARD` | 每組目標房間數 | `50` |
| `SUBCHECK_TICK_SECS` | 目標分組觸發間隔（秒） | `10` |
| `SUBCHECK_MIN_INTERVAL_SECS` | 完整輪詢週期下限（秒） | `60` |
| `SUBCHECK_MAX_INTERVAL_SECS` | 完整輪詢週期上限（秒） | `300` |
| `SUBCHECK_MAX_SHARDS` | 分組數上限 | `32` |
| `SUBCHECK_JITTER_SECS` | 每組輪詢觸發前的最大隨機延遲（秒），打散固定節奏、降低被 B 站反爬識別風險 | `2` |

訂閱極多且仍感到 CPU 尖峰偏高時，可略增大 `SUBCHECK_ROOMS_PER_SHARD`（每組更多房間、分組更少）或略增大 `SUBCHECK_MIN_INTERVAL_SECS`（接受稍慢的開播檢測）。若頻繁觸發風控或日誌出現切換帳號憑證模式，可適當增大 `SUBCHECK_JITTER_SECS`。使用流程見 [錄製直播間](/zh-tw/guides/recording/#訂閱房間)。

## 直播間信息查詢（風控自動切換）

查詢房間是否在播、拉取房間標題封面等信息時（包括訂閱輪詢、Web 介面手動刷新、開始錄製前的狀態檢查），程式**預設先用匿名方式**向 B 站請求，不攜帶你的登入 Cookie，以降低帳號被風控的風險。

當 B 站返回表示「當前網路或請求被限制」的 API 錯誤碼時（例如常見的 `-352` 等，HTTP 仍可能是 200），程式會**自動改用已登入帳號的 Cookie 重試同一次查詢**，並在日誌中提示：

- **警告**：首次觸發切換時，例如「直播間信息請求觸發風控，已切換為帳號憑證模式」
- **信息**：連續多次觸發後鎖定為帳號模式，或探測恢復匿名模式時的提示

你**無需設定**也**不會在 Web 介面看到橫幅**；若關心狀態，查看後端日誌即可。

| 說明 | 內容 |
|------|------|
| 適用請求 | 所有「直播間基礎信息」類查詢（與訂閱輪詢共用同一機制） |
| 需已登入 | `BILIBILI_LOGIN_MODE=controller` 或 `startup`，且已完成 B 站掃碼登入；否則風控時無法自動切換，只能等網路恢復 |
| `anonymous` 模式 | 全程匿名，無 Cookie 可切換；被風控時查詢可能失敗直至 IP 限制解除 |
| 帳號安全 | 僅在匿名請求被判定為風控時才使用 Cookie；正常網路下訂閱輪詢不消耗帳號配額 |
| 恢復 | 進入帳號模式後，約每 5 分鐘用匿名方式探測一次；連續探測成功後會自動回到匿名模式 |

明確屬於「房間不存在」、參數錯誤（`400` / `404`）或服務端錯誤（`500` 及以上）的響應**不會**觸發 Cookie 重試。
