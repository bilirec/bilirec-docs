遇到報錯時，先看介面或日誌裡的那句中文；若已經說清楚了，按提示操作即可。下面只整理**容易誤會、或介面上看不出來原因**的情況。

## 後端錄製

### 開了內網穿透和 HTTPS 後，程式啟動就報錯？

**你會看到：** 一啟動就退出，日誌裡提到 FRP 或 HTTPS 設定不對。

**為什麼：** 內網穿透連回本機的協議，需與你本機是否開 HTTPS 一致。兩邊開關對不上，程式會拒絕啟動，避免連上卻用不了。

**你可以：**
1. 若本機**沒有**配證書，內網穿透用 HTTP 方式連回來（常見家用場景）。
2. 若本機**已經**配了證書走 HTTPS，內網穿透也要用 HTTPS 方式連回來。
3. 具體變數組合見 [遠端存取（FRP）](/zh-tw/configuration/frp/)。

### FRP 伺服器地址填錯了？

**你會看到：** 日誌裡寫「無效的 FRP_SERVER 格式」，或隧道一直連不上。

**為什麼：** 地址一般宜寫成 `主機名或IP:端口`，例如 `tunnel.bilirec.org:7000`。只寫域名時，程式會預設用 7000 端口；帶 `http://` 的網址可能無法正確解析。

**你可以：** 改成 `地址:端口` 後重啟；自架 FRP 時還要填好 Token（見下一條）。

### 用自己的 FRP 伺服器，外網還是進不來？

**你會看到：** 程式能啟動，但日誌裡 FRP 初始化失敗，或沒有公網地址。

**為什麼：** 連**自己的** FRP 時通常要填認證 Token；用官方公共服務時，只開 `FRP_ENABLED=true` 即可。

**你可以：**
1. 確認 `FRP_SERVER`、`FRP_BASE_DOMAIN` 與伺服器一致。
2. 自架服務填上 `FRP_TOKEN`。
3. 詳見 [遠端存取（FRP）](/zh-tw/configuration/frp/)。

### 日誌裡出現「觸發風控」「已切換為帳號憑證模式」？

**你會看到：** 後端日誌有警告或信息，例如「直播間信息請求觸發風控，已切換為帳號憑證模式」「已恢復匿名模式」。Web 介面沒有額外提示，訂閱與錄製通常仍正常。

**為什麼：** 當前網路的匿名查詢被 B 站臨時限制時，程式會自動用你已登入的 Cookie 重試直播間信息查詢，以免漏檢開播或無法開始錄製。

**你可以：**
1. **一般無需處理**；網路恢復後程式會定期探測並自動回到匿名模式。
2. 若長期反覆出現，可能是本機 IP 被 B 站限流較嚴，可減少訂閱數量或略增大 [訂閱檢查週期](/zh-tw/configuration/recording/#訂閱房間狀態檢查)。
3. 使用 `BILIBILI_LOGIN_MODE=anonymous` 時無法切換，需改為 `controller` 並完成掃碼登入，或等待 IP 限制解除。若仍頻繁觸發，可略增大 `SUBCHECK_JITTER_SECS` 打散輪詢節奏。詳見 [認證與安全 · 直播間信息查詢與風控](/zh-tw/configuration/auth-security/#直播間信息查詢與風控)。

### 同一場直播裡為什麼會出現「標題-1」「標題-2」多個檔案？

**你會看到：** 目錄裡除了 `標題-時間戳.flv`，還有 `標題-時間戳-1.flv`、`標題-時間戳-2.flv` 等同一場、**同一時間戳**下的分段。

**為什麼：** 這是**直播輪轉**（例如 PK、換畫質導致流頭變化）時程式自動切的新檔案，屬於正常設計。這和斷網重連開的「新時間戳檔案」不是一回事。

**你可以：** 按檔案名順序觀看即可。若你想了解輪轉規則，見 [錄製指南](/zh-tw/guides/recording/)。

### 斷網重連後怎麼又多了一個不同時間戳的檔案？

**你會看到：** 網路卡了一下之後，目錄裡多了一個**新的時間戳**檔案（檔案名裡沒有 `-1`、`-2` 這種後綴）。

**為什麼：** 斷線後程式會重連並**新開一個檔案**繼續錄，避免把兩段不連續的流硬拼在一起，也不會覆蓋舊檔案。`RECORDING_RECOVERY_DURATION` 的 `preserve` / `reset` 只影響「還能錄多久」，**不影響**檔案名：兩種模式都是新時間戳開檔。

**你可以：** 按時間戳區分各段即可；若經常斷網，先檢查網路。重連相關設定見 [錄製設定](/zh-tw/configuration/recording/)。

### 開了「錄完轉 MP4」，卻一直不轉？

**你會看到：** 錄製結束了，列表裡仍是 flv/ts，或轉碼任務很久不動。

**為什麼：** 預設為了省 CPU，**正在錄別的房間時**不會跑本地轉碼；機器上也可能沒裝 ffmpeg，或沒配雲端轉碼。

**你可以：**
1. 等所有錄製都停掉後再看是否開始轉。
2. 確認已安裝 ffmpeg，或設定了 CloudConvert。
3. 若要在錄製時也轉，需在設定裡單獨打開（見 [轉換設定](/zh-tw/configuration/conversion/)）。

### 轉碼老是失敗、反覆重試？

**你會看到：** 轉換任務一會兒失敗一會兒又排隊。

**為什麼：** 常見原因有兩類：一是轉出來的 MP4 **太小**（不到 1MB 或不到原檔案一半），程式會認為結果不可靠而重試；二是**源檔案本身不完整或已損壞**（錄製中斷、磁碟問題等），怎麼轉都難成功。

**你可以：** 先確認源 flv/ts 能否正常播放；源檔案有問題時只能保留原格式或重新錄。詳見 [轉換設定](/zh-tw/configuration/conversion/)。

### 大檔案走 CloudConvert 創建任務失敗？

**你會看到：** 日誌裡和「公開地址」「預簽名」有關的錯誤。

**為什麼：** 雲端要回來拉你的檔案，需要填一個**能從外網存取到的完整網址**（`PUBLIC_BASE_URL`）；留空或只寫內網地址時，CloudConvert 無法拉取檔案。

**你可以：** 在設定裡設好對外的 `http://` 或 `https://` 地址，或改用本機 ffmpeg 轉小檔案。見 [轉換設定](/zh-tw/configuration/conversion/)。

## Web 管理介面

### 打開 app.bilirec.org，卻連不上自己的錄製服務？

**你會看到：** 頁面能開，但登入或列表一直失敗。

**為什麼：** `app.bilirec.org` 是 **HTTPS 網頁**，不是你的錄製程式本身。網頁要單獨填後端地址才能連上。由於瀏覽器安全限制，從 HTTPS 頁面去連 **HTTP 局域網地址**（如 `http://192.168.x.x:8080`）往往會被攔下；**HTTP 後端一般只能填本機** `http://localhost:8080` 或 `http://127.0.0.1:8080`。亦有可能是瀏覽器沒有打開本地網路存取權限。

**你可以：**
1. 後端跑在本機時，用 `http://localhost:8080` 或 `http://127.0.0.1:8080`。
2. 檢查瀏覽器是否打開了本地網路存取權限或者瀏覽器是否支持局域網存取。
3. 後端在局域網另一台機器上時，需要給後端配 **HTTPS**，或在同一台機器上用瀏覽器打開管理頁；HTTPS 的 app.bilirec.org 無法直接連家裡的 HTTP 地址。

### 用 app.bilirec.org 連遠端後端，一直提示「會話已過期」？

**你會看到：** 帳號密碼明明正確，登入後剛進主介面幾秒就彈出「會話已過期，請重新登入」；怎麼重登都一樣。後端本身可能一直正常。

**為什麼：** [app.bilirec.org](https://app.bilirec.org/) 和你的後端地址（FRP 的 `https://xxx.tunnel.bilirec.org`、自架的 `https://api.example.com` 等）是**兩個不同網址**。預設設定下，瀏覽器往往記不住登入狀態，介面會先顯示成功，幾秒後又讓你重登。若使用 FRP 且 bilirec 重啟過，子域名可能已變，舊地址上的登入也會失效。

**根域名不同時**（例如自架前端 `https://bilirec.example.com` 連 FRP 的 `https://xxx.tunnel.bilirec.org`），瀏覽器還可能把登入狀態當作第三方 Cookie 攔下；**app.bilirec.org 連 `*.tunnel.bilirec.org` 同屬 `bilirec.org`，一般不會有這個問題**，優先檢查 `PRODUCTION_MODE` 即可。

**你可以：**
1. 在後端設定 `PRODUCTION_MODE=true`，並設定 `USERNAME`、`PASSWORD`、固定的 `JWT_SECRET`；`FRONTEND_URL` 填你實際使用的管理頁地址（官方頁用 `https://app.bilirec.org`，自架前端填你的 HTTPS 域名），然後重啟 bilirec。
2. 在登入頁填入**當前**的後端地址（FRP 每次重啟子域名可能變化；自架域名一般不變）。
3. **僅當前端與後端根域名不同**（自架前端 + FRP 等）且仍登入不上時：在瀏覽器裡**允許第三方 Cookie**，或將前端與後端域名加入例外。Chrome / Edge：「設定 → 隱私和安全 → 第三方 Cookie」；Safari：關閉「阻止跨站跟蹤」；Firefox：對前端域名放寬隱私保護。
4. 本機 `localhost` 開發不受此影響，無需改 `PRODUCTION_MODE`。

詳見 [遠端存取（FRP）— 通過 app.bilirec.org 登入](/zh-tw/guides/frp/#通過-appbilirecorg-登入)（自架 HTTPS 後端同樣適用）與 [認證與安全](/zh-tw/configuration/auth-security/#何時需要生產模式)。

### 收不到開播推送通知？

**你會看到：** 瀏覽器或 PWA 裡從沒彈過通知。

**為什麼：** 需要允許通知權限、成功連過後端，只讀帳號也不會推送。若用的是 **PWA 或 Chrome**，在內地網路下還可能因為**連不上 Google FCM** 而收不到推送。

**你可以：**
1. 在瀏覽器設定裡允許本站通知。
2. 確認登入頁的伺服器地址正確並已保存。
3. 用管理員帳號訂閱推送；內地使用 PWA 或 Chrome 的用戶若仍無推送，可改用 SSE 等方案。詳見 [Web 介面](/zh-tw/guides/web-ui/) 與 [通知設定](/zh-tw/configuration/notifications/)。

## Android

### 錄製保存到 SD 卡後變慢？

**你會看到：** 換到 SD 卡路徑後，錄製或介面變卡。

**為什麼：** 外置 SD 通常比手機內置儲存慢。若輸出路徑在 microSD 卡上，建議在 App **儲存策略**裡開啟 **microSD 卡磨損保護**。

**你可以：** 在 App 的儲存相關設定裡按需開關；追求速度可改回內置儲存。見 [Android 指南](/zh-tw/guides/android/) 與 [Android 預設設定](/zh-tw/configuration/android-defaults/)。

### 啟動後切到後台，過一會兒服務就自己停了？

**你會看到：** 剛還能錄，划走 App 或鎖屏一會兒，通知欄服務沒了或 Web 連不上。

**為什麼：** 最常見是**沒關電池優化**（或系統殺後台）。Android 對後台和省電管得很嚴，不按提示放行就容易被停掉。

**你可以：**
1. 按 App 提示**關閉電池優化**、允許通知（前臺服務需要）。
2. 若選了外置儲存路徑，按提示授予儲存權限。
3. 建議避免讓系統「清理後台」。見 [Android 指南](/zh-tw/guides/android/)。
4. **國產手機**（華為、OPPO 等）還需完成額外的系統設定。見 [國產手機後台設定](/zh-tw/guides/android-mainland/)。

### 國產手機上後台經常超時或斷連？

**你會看到：** 用 Wi‑Fi 時較穩定，換成移動數據後開播檢測變慢、錄製拉流超時，或鎖屏後通知欄狀態長時間不更新。

**為什麼：** 大陸國產系統對後台進程的限制比一般 Android 更嚴；華為等機型在使用移動數據時還會額外限制後台聯網。

**你可以：**
1. 按 [國產手機後台設定](/zh-tw/guides/android-mainland/) **從第一步起逐步嘗試**；每做完一步先測試，問題解決即可停止。
2. 外出使用移動數據時，可暫時連接 Wi‑Fi 或熱點以恢復流暢通訊。

### 用 bilirec-mobile 打開 PWA 後，連不上本機後端？

**你會看到：** 預設地址連不上，或換瀏覽器才好使。

**為什麼：** 連不上更常見是**瀏覽器限制**：部分瀏覽器（例如 Opera）對存取本機服務支持不好，換 Chrome 等通常能解決。

**你可以：**
1. 優先用 Chrome 打開 PWA；伺服器地址用預設的 `http://localhost:8080` 或 `http://127.0.0.1:8080` 即可。
2. 確認 bilirec-mobile 裡後端已成功啟動。

### 打開錄製程式後還要填地址？

**你會看到：** 從 App 跳進網頁，擔心還要再填伺服器。

**為什麼：** App 負責啟動手機裡的後端；管理頁是另一個程式。正常來說登入介面可以直接點擊登入（預設就是本地地址），之後再也不需要填地址了。

**你可以：** 進入登入介面後直接點擊登入，或安裝為PWA之後直接打開PWA啟動。

## 性能與容量規劃

### CPU 突然飆高怎麼辦？

**你會看到：** 任務管理器或手機通知欄裡 CPU 百分比偶爾衝高。

先對照下表判斷尖峰來源，再調對應環境變數：

| 可能來源 | 你怎麼分辨 | 相關環境變數 | 建議方向 |
| -------- | ---------- | ------------ | -------- |
| **寫盤 / flush 尖峰** | 正在錄製時週期性衝高；microSD、機械盤更明顯 | `LIVE_STREAM_WRITER_BUFFER_SIZE`<br />`LIVE_STREAM_WRITER_FLUSH_PERIOD_SECS`<br />`LIVE_STREAM_WRITER_SYNC_PERIOD_SECS` | 適當加大合併緩衝、拉長 flush 間隔；**保持** `SYNC_PERIOD=0`（開週期性整檔案落盤會更吃 CPU） |
| **Go 頻繁 GC（Docker）** | 容器記憶體很空但 CPU 仍週期性抖；多發生在 limit 很大卻未改 `GOMEMLIMIT` | `GOMEMLIMIT` | 把 `GOMEMLIMIT` 調到約為容器 limit 的 75%～85%；見 [Docker 指南](/zh-tw/guides/docker/#go-記憶體上限gomemlimit) |
| **多路搶同一張盤** | 多路同時錄到同一張 SD / HDD 時尖峰疊加 | `SEQUENTIAL_WRITE` | 同一張卡上多路錄製時保持 `true`（Android 輸出到 microSD 時在 App 開啟 **microSD 卡磨損保護**） |
| **訂閱開播輪詢** | **閒置**時大約每隔固定秒數有小尖峰 | `SUBCHECK_ROOMS_PER_SHARD`<br />`SUBCHECK_MIN_INTERVAL_SECS` | 預設已按訂閱規模自動調度；仍偏高可略增大每組房間數或週期下限 |
| **後台轉碼** | 轉碼任務進行中 CPU 持續偏高 | `FFMPEG_MAX_CONCURRENT_TASKS`<br />`FFMPEG_ALLOW_DURING_RECORDING`<br />`FFMPEG_CHECK_INTERVAL_SECS` | 建議避免邊錄邊轉；並發轉碼可保持 `1`；詳見 [轉換設定](/zh-tw/configuration/conversion/) |

**你可以：**
1. 樹莓派 + microSD：從 [I/O 調優](/zh-tw/configuration/io-tuning/) 與 [樹莓派預設設定](/zh-tw/configuration/pi5-defaults/) 調整寫盤相關項。
2. 訂閱房間特別多：見 [錄製設定 · 訂閱房間狀態檢查](/zh-tw/configuration/recording/#訂閱房間狀態檢查)（預設自動調度，一般無需隨訂閱改環境變數）。
3. 轉碼與錄製錯峰：見 [轉換設定](/zh-tw/configuration/conversion/)。

### HDD NAS 用 ZFS，CPU 比平時高一點？

**你會看到：** 同樣路數，輸出目錄在機械盤 **ZFS 池** 上時，CPU 比 EXT4/XFS 等同場景多幾個百分點，但錄製仍順暢。

**為什麼：** ZFS 要做校驗、元數據與寫入合併，多路錄製時這部分開銷會疊加。這是 **ZFS 本身的代價**，與是否設 `SEQUENTIAL_WRITE=false` **無直接關係**。

**你可以：**
1. 一般不必處理，屬正常現象。
2. 若是**週期性尖峰**或 NAS 明顯變卡，再對照上一節 [CPU 突然飆高](/zh-tw/guides/faq/#cpu-突然飆高怎麼辦) 排查寫盤、多路搶盤等項。
3. ZFS 池調參見 [HDD/NAS 調優](/zh-tw/configuration/hdd-nas-tuning/)。

### Docker 記憶體明明很空，卻頻繁 GC？

**你會看到：** 容器 memory limit 設得很大（如 2GB、4GB），`docker stats` 裡用量還不到一半，但 CPU 週期性抖動，錄製偶爾微卡；調高 `MAX_CONCURRENT_RECORDINGS` 後更明顯。**僅 Docker 部署**會遇到；二進制直跑、Android 無鏡像內建的 `GOMEMLIMIT` 預設值。

**為什麼：** 官方 **Docker 鏡像**預設 `GOMEMLIMIT=768MiB`（其它部署方式沒有寫死）。Go 按這個數自律堆記憶體，**不會**因為你給容器分了更多 limit 就自動多用。堆快碰到 768MiB 就會勤 GC，而容器監控裡的「空餘」對 Go 堆不一定可用。

**你可以：**
1. 用 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/) 估路數需求，再設容器 memory limit。
2. 用 `-e GOMEMLIMIT=...` 把它調到**略低於** limit（約 75%～85%），與 limit 一起放大。詳見 [Docker 指南 · GOMEMLIMIT](/zh-tw/guides/docker/#go-記憶體上限gomemlimit)。
3. 若 limit 本就約 1GB、只錄少數路，預設 `768MiB` 一般不必改。

### 錄製時記憶體佔用很高怎麼辦？

**你會看到：** 監控裡記憶體在錄播時緩緩上漲。

**為什麼：** 常見是 **程式本身的緩衝**（路數越多、4K 越高越大）或 **系統頁緩存**（檔案被緩存到記憶體裡）。先對照 [記憶體佔用估算 · 監控裡的記憶體從哪來](/zh-tw/configuration/memory-estimation/#監控裡的記憶體從哪來) 判斷來源。

**你可以：**
1. 若是程式本身的緩衝：按 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/) 調整 `READ_STREAM_*`、`LIVE_STREAM_WRITER_*`（通常比直接減路數更合適）。
2. 若是系統頁緩存：預設已開冷釋放與關檔釋放；仍偏高見 [I/O 調優](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android) 或 [常見問題 · 監控記憶體偏高](/zh-tw/guides/faq/#監控記憶體偏高兩個相關選項怎麼理解)。
3. 只有確認設備**確實達不到**公式或實測要求時，再考慮減少 `MAX_CONCURRENT_RECORDINGS`。

### 怎麼知道我的設備能錄幾路？

**你會看到：** 不確定該設多少路、容器該分多少記憶體。

**為什麼：** 上限取決於記憶體公式、CPU、以及磁碟（尤其是 microSD）三者裡**最吃緊**的那一項。

**你可以：**
1. 用 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/) 算一下目標路數需要多少記憶體，容器記憶體留約兩成餘量。
2. 對照 [性能實測](/zh-tw/guides/performance-benchmark/) 裡相近設備的數據。
3. 從 **1 路**開始試，逐步加到穩定為止。樹莓派 + SD 卡還要參考 [樹莓派預設設定](/zh-tw/configuration/pi5-defaults/)。

### 都停錄、轉碼也結束了，記憶體為什麼還不少？

**你會看到：** 任務都停了，Docker 或手機通知欄裡記憶體仍比剛啟動時高（例如五十～幾百 MB）。

**為什麼：**

1. **記憶體池預熱** — bilirec 會把讀入、寫盤用的記憶體塊留在進程裡複用，減輕下次錄製時的分配開銷。停錄後常見幾十 MB 的「底座」，見 [性能實測「恢復閒置」](/zh-tw/guides/performance-benchmark/)（約 54 MB）。錄過 4K / 杜比時，高畫質那套池子大約 **15 分鐘**不用後會自動釋放。
2. **系統頁緩存** — 數據寫入磁碟後，Linux / Android 往往還會在記憶體裡留一份檔案副本（頁緩存），用來加速以後的讀寫。Docker 或手機監控的「總記憶體」會把這份算進去。

**你可以調：**

| 想改善什麼 | 環境變數 | 方向 |
| ---------- | -------- | ---- |
| 讓系統忘掉檔案頁緩存 | `DROP_FILE_PAGE_CACHE` | 保持 `true`（預設已開）；關檔 / 轉碼後整份釋放 |
| 錄製中更勤地清舊頁緩存 | `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | 改小間隔（如 `60` → `30`）；`0` = 關閉錄製中冷釋放 |
| 降低記憶體池預熱佔用 | `LIVE_STREAM_WRITER_CHAN_BUFFER_SIZE`<br />`READ_STREAM_CHAN_BUFFER_SIZE`<br />`LIVE_STREAM_WRITER_BUFFER_SIZE` 等 | 略減小緩衝，見 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/#想省記憶體優先改哪幾個) |

冷釋放細節（何時才真正執行、與整檔案落盤的區別）見 [I/O 調優 · 錄製檔案緩存釋放](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android)。

**何時要擔心：** 停錄很久仍**持續上漲**或明顯卡死 — 再查日誌；多數情況下屬於上面三類正常殘留。

### 在線播放或下載後，監控記憶體一直很高？

瀏覽器在線播放、或下載已錄好的 MP4 時，系統會為讀盤加速在記憶體裡**再留一份檔案副本**（與錄製寫入類似，還是屬於系統頁緩存）。播得越久、拖進度條越頻繁，這份緩存可能越大；**停播後**若監控仍居高不下，多半是緩存還沒被系統收回。

**預設行為（`DROP_FILE_PAGE_CACHE=true`）：** 最後一次播放或下載請求結束後，會等待 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` 秒（預設 **60**）再釋放該檔案的緩存；這段時間內若再次播放同一檔案，計時會重新算起。若你把該變數設為 `0`（僅關閉**錄製中**冷釋放），播放/下載空閒仍按預設 **60** 秒等待後釋放，避免與關檔釋放（`DROP_FILE_PAGE_CACHE`）綁死。

**侷限：** 長時間連續播放、Range 請求不斷到來時，計時會被反覆重置，**播放過程中**監控仍可能上漲；本機制主要改善**停播 / 下載完成後**曲線不降的問題。

**若更在意播放流暢、不在意監控數字：** 設 `DROP_FILE_PAGE_CACHE=false`（同時也會影響關檔釋放，見下節）。

### 監控記憶體偏高？兩個相關選項怎麼理解？

**預設已開啟**（一般不用改）：

| 環境變數 | 說明 | 預設 |
| -------- | ------ | ---- |
| `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | **錄製過程中**，每隔一段時間讓系統釋放「已經寫好、暫時不會再讀」的舊檔案緩存 | `60`（秒） |
| `DROP_FILE_PAGE_CACHE` | **一段錄完或轉碼結束後**，釋放整份檔案的系統頁緩存 | `true` |

兩者配合，可避免 Docker / 手機監控裡記憶體隨錄製一路頂到上限。代價是：剛錄完立刻播放、或錄製中回放已寫好的前半段，可能要從磁碟重新讀，略慢。

**冷釋放不是「每 60 秒必清一次」：** 單段檔案要長到約 96 MB 以上才會開始清理舊前綴；短 HLS 分段主要靠關檔時的 `DROP_FILE_PAGE_CACHE`。詳見 [I/O 調優 · 錄製檔案緩存釋放](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android)。

**適合保持預設：** 樹莓派、1GB 容器、Android 後台、很少錄完馬上拖進度條。

**建議關閉（回放優先）：** SSD / NAS、錄完馬上要秒開播放。設 `DROP_FILE_PAGE_CACHE=false` 且 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=0`。

**和「週期性整檔案落盤」(`LIVE_STREAM_WRITER_SYNC_PERIOD_SECS`) 的區別：**

- `LIVE_STREAM_WRITER_SYNC_PERIOD_SECS` 只是為了**斷電少丟數據**，錄製中反覆把整個檔案刷進卡，**很傷 microSD**，也**壓不住**監控記憶體。
- 兩個都設定了：以整檔案落盤為準，不再單獨按 60 秒清緩存；**僅當 `COLD_CACHE_RELEASE_SECS>0`** 時，每次落盤成功後才順便釋放當前頁緩存。
- `LIVE_STREAM_WRITER_SYNC_PERIOD_SECS` **本身不會自動釋放緩存**：它只負責落盤；釋放緩存靠落盤後的頁緩存釋放、需要配合設定 `DROP_FILE_PAGE_CACHE=true` 才能生效。

若日誌出現 `釋放檔案頁緩存失敗` 或 `冷緩存釋放失敗`，錄製通常仍可繼續，只是記憶體可能降不下來；見 [I/O 調優 · 錄製檔案緩存釋放](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android)。
