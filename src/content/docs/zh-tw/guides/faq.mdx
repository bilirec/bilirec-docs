遇到報錯時，先看介面或日誌裡的那句中文；若已經說清楚了，按提示操作即可。下面只整理**容易誤會、或介面上看不出來原因**的情況。

## 後端錄製

### 開了內網穿透和 HTTPS 後，程序啟動就報錯？

**你會看到：** 一啟動就退出，日誌裡提到 FRP 或 HTTPS 設定不對。

**為什麼：** 內網穿透連回本機的協議，必須和你本機是否開 HTTPS 一致。兩邊開關對不上，程序會拒絕啟動，避免連上卻用不了。

**你可以：**
1. 若本機**沒有**配證書，內網穿透用 HTTP 方式連回來（常見家用場景）。
2. 若本機**已經**配了證書走 HTTPS，內網穿透也要用 HTTPS 方式連回來。
3. 具體變量組合見 [遠程訪問（FRP）](/zh-tw/configuration/frp/)。

### FRP 服務器地址填錯了？

**你會看到：** 日誌裡寫「無效的 FRP_SERVER 格式」，或隧道一直連不上。

**為什麼：** 地址一般應寫成 `主機名或IP:端口`，例如 `tunnel.bilirec.org:7000`。只寫域名時，程序會默認用 7000 端口；不要寫成帶 `http://` 的網址。

**你可以：** 改成 `地址:端口` 後重啟；自架 FRP 時還要填好 Token（見下一條）。

### 用自己的 FRP 服務器，外網還是進不來？

**你會看到：** 程序能啟動，但日誌裡 FRP 初始化失敗，或沒有公網地址。

**為什麼：** 連**自己的** FRP 時通常要填認證 Token；用官方公共服務時，只開 `FRP_ENABLED=true` 即可。

**你可以：**
1. 確認 `FRP_SERVER`、`FRP_BASE_DOMAIN` 與服務器一致。
2. 自架服務填上 `FRP_TOKEN`。
3. 詳見 [遠程訪問（FRP）](/zh-tw/configuration/frp/)。

### 同一場直播裡為什麼會出現「標題-1」「標題-2」多個檔案？

**你會看到：** 目錄裡除了 `標題-時間戳.flv`，還有 `標題-時間戳-1.flv`、`標題-時間戳-2.flv` 等同一場、**同一時間戳**下的分段。

**為什麼：** 這是**直播輪轉**（例如 PK、換畫質導致流頭變化）時程序自動切的新檔案，屬於正常設計，不是壞了。

**你可以：** 按檔案名順序觀看即可。若你想了解輪轉規則，見 [錄製指南](/zh-tw/guides/recording/)。

### 斷網重連後怎麼又多了一個不同時間戳的檔案？

**你會看到：** 網絡卡了一下之後，目錄裡多了一個**新的時間戳**檔案（檔案名裡沒有 `-1`、`-2` 這種後綴）。

**為什麼：** 斷線後程序會重連並**新開一個檔案**繼續錄，避免把兩段不連續的流硬拼在一起。這和上一節說的「同時間戳帶 -1/-2」不是一回事。

**你可以：** 按時間戳區分各段即可；若經常斷網，先檢查網絡。重連相關設定見 [錄製指南](/zh-tw/guides/recording/)。

### 開了「錄完轉 MP4」，卻一直不轉？

**你會看到：** 錄製結束了，列表裡仍是 flv/ts，或轉碼任務很久不動。

**為什麼：** 默認為了省 CPU，**正在錄別的房間時**不會跑本地轉碼；機器上也可能沒裝 ffmpeg，或沒配雲端轉碼。

**你可以：**
1. 等所有錄製都停掉後再看是否開始轉。
2. 確認已安裝 ffmpeg，或設定了 CloudConvert。
3. 若要在錄製時也轉，需在設定裡單獨打開（見 [轉換設定](/zh-tw/configuration/conversion/)）。

### 轉碼老是失敗、反覆重試？

**你會看到：** 轉換任務一會兒失敗一會兒又排隊。

**為什麼：** 常見原因有兩類：一是轉出來的 MP4 **太小**（不到 1MB 或不到原檔案一半），程序會認為結果不可靠而重試；二是**源檔案本身不完整或已損壞**（錄製中斷、磁盤問題等），怎麼轉都難成功。

**你可以：** 先確認源 flv/ts 能否正常播放；源檔案有問題時只能保留原格式或重新錄。詳見 [轉換設定](/zh-tw/configuration/conversion/)。

### 大檔案走 CloudConvert 創建任務失敗？

**你會看到：** 日誌裡和「公開地址」「預簽名」有關的錯誤。

**為什麼：** 雲端要回來拉你的檔案，需要填一個**能從外網訪問到的完整網址**（`PUBLIC_BASE_URL`），不能留空或只寫內網地址。

**你可以：** 在設定裡設好對外的 `http://` 或 `https://` 地址，或改用本機 ffmpeg 轉小檔案。見 [轉換設定](/zh-tw/configuration/conversion/)。

## Web 管理介面

### 打開 app.bilirec.org，卻連不上自己的錄製服務？

**你會看到：** 頁面能開，但登錄或列表一直失敗。

**為什麼：** `app.bilirec.org` 是 **HTTPS 網頁**，不是你的錄製程序本身。網頁要單獨填後端地址才能連上。由於瀏覽器安全限制，從 HTTPS 頁面去連 **HTTP 局域網地址**（如 `http://192.168.x.x:8080`）往往會被攔下；**HTTP 後端一般只能填本機** `http://localhost:8080` 或 `http://127.0.0.1:8080`。

**你可以：**
1. 後端跑在本機時，用 `http://localhost:8080` 或 `http://127.0.0.1:8080`。
2. 後端在局域網另一臺機器上時，需要給後端配 **HTTPS**，或在同一臺機器上用瀏覽器打開管理頁，不能指望 HTTPS 的 app.bilirec.org 直接連家裡的 HTTP 地址。

### 收不到開播推送通知？

**你會看到：** 瀏覽器或 PWA 裡從沒彈過通知。

**為什麼：** 需要允許通知權限、成功連過後端，並且用**已保存服務器地址**的方式打開過管理頁；只讀賬號也不會推送。若用的是 **PWA 或 Chrome**，在內地網絡下還可能因為**連不上 Google FCM** 而收不到推送（與 bilirec 設定無關）。

**你可以：**
1. 在瀏覽器設置裡允許本站通知。
2. 確認登錄頁的服務器地址正確並已保存。
3. 用管理員賬號訂閱推送；內地使用 PWA 或 Chrome 的用戶若仍無推送，可改用 SSE 等方案。詳見 [Web 介面](/zh-tw/guides/web-ui/) 與 [通知設定](/zh-tw/configuration/notifications/)。

## Android

### 錄製保存到 SD 卡後變慢？

**你會看到：** 換到 SD 卡路徑後，錄製或介面變卡。

**為什麼：** 外置 SD 通常比手機內置存儲慢。是否在 SD 上啟用「排隊寫盤」等策略，請在 **bilirec-mobile App 設置**裡調整（存儲策略 / 序列化寫入），不再由後端自動改。

**你可以：** 在 App 的存儲相關設置裡按需開關；追求速度可改回內置存儲。見 [Android 指南](/zh-tw/guides/android/) 與 [Android 默認設定](/zh-tw/configuration/android-defaults/)。

### 啟動後切到後臺，過一會兒服務就自己停了？

**你會看到：** 剛還能錄，划走 App 或鎖屏一會兒，通知欄服務沒了或 Web 連不上。

**為什麼：** 最常見是**沒關電池優化**（或系統殺後臺）。Android 對後臺和省電管得很嚴，不按提示放行就容易被停掉。

**你可以：**
1. 按 App 提示**關閉電池優化**、允許通知（前臺服務需要）。
2. 若選了外置存儲路徑，按提示授予存儲權限。
3. 儘量別讓系統「清理後臺」。見 [Android 指南](/zh-tw/guides/android/)。

### 用 bilirec-mobile 打開 PWA 後，連不上本機後端？

**你會看到：** 默認地址連不上，或換瀏覽器才好使。

**為什麼：** `localhost` 和 `127.0.0.1` **指的是同一臺手機**，並不是「必須用 127.0.0.1 才行」。連不上更常見是**瀏覽器限制**：部分瀏覽器（例如 Opera）對訪問本機服務支持不好，換 Chrome 等通常能解決。

**你可以：**
1. 優先用 Chrome 打開 PWA；服務器地址用默認的 `http://localhost:8080` 即可（與 `http://127.0.0.1:8080` **等價**，不必特意改）。
2. 確認 bilirec-mobile 裡後端已成功啟動。

### 打開錄製程序後還要填地址？

**你會看到：** 從 App 跳進網頁，擔心還要再填服務器。

**為什麼：** App 負責啟動手機裡的後端；管理頁是另一個程序。好消息是：**默認就是 `http://localhost:8080`**，一般**不用手改**，直接點登錄即可（前提是後端已在 App 裡啟動成功）。

**你可以：** 打開 PWA 後先點登錄；只有連不上時再檢查地址或換瀏覽器（見上一條）。

## 性能與容量規劃

### CPU 突然飆高怎麼辦？

**你會看到：** 任務管理器或手機通知欄裡 CPU 百分比偶爾衝高。

**為什麼：** 很多時候不是「算力不夠」，而是**寫盤太勤**——尤其 microSD、機械硬盤上，頻繁 flush/sync 會造成 CPU 和 I/O 尖峰。訂閱房間多、後臺轉碼也會疊加一些佔用。閒置時若大約每 10 秒有小尖峰，可能來自訂閱開播輪詢（subcheck）；可調 `SUBCHECK_CHECK_INTERVAL_SECS` / `SUBCHECK_SHARD_COUNT`（見 [錄製設定](/zh-tw/configuration/recording/)）。

**你可以：**
1. 樹莓派 + microSD：適當**加大寫盤緩衝**、**拉長寫盤間隔**、必要時關閉錄製中整檔案落盤（見 [I/O 調優](/zh-tw/configuration/io-tuning/) 與 [樹莓派默認設定](/zh-tw/configuration/pi5-defaults/)）。
2. 訂閱房間特別多時，把檢查分組調多、一輪時間略拉長（見 [錄製設定](/zh-tw/configuration/recording/)）。
3. 儘量不要邊錄邊轉碼，或把轉碼挪到空閒時段。

### 錄製時記憶體佔用很高怎麼辦？

**你會看到：** 監控裡記憶體在錄播時明顯上漲。

**為什麼：** 直播數據會在記憶體裡先排隊再寫盤，路數越多、畫質越高（4K），佔用越大。

**你可以：**
1. 按 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/) 調整緩衝相關設定（通常比直接減路數更合適）。
2. 不要只看 Docker 總記憶體——系統緩存檔案也會讓數字偏大。
3. 只有確認設備**確實達不到**公式或實測要求時，再考慮減少同時錄製路數。

### 怎麼知道我的設備能錄幾路？

**你會看到：** 不確定該設多少路、容器該分多少記憶體。

**為什麼：** 上限取決於記憶體公式、CPU、以及磁盤（尤其是 microSD）三者裡**最吃緊**的那一項。

**你可以：**
1. 用 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/) 算一下目標路數需要多少記憶體，容器記憶體留約兩成餘量。
2. 對照 [性能實測](/zh-tw/guides/performance-benchmark/) 裡相近設備的數據。
3. 從 **1 路**開始試，逐步加到穩定為止。樹莓派 + SD 卡還要參考 [樹莓派默認設定](/zh-tw/configuration/pi5-defaults/)。

### 都停錄、轉碼也結束了，記憶體為什麼還不少？

**你會看到：** 任務都停了，Docker 或手機通知欄裡記憶體仍比剛啟動時高（例如五十～幾百 MB）。

**為什麼：**

1. **程序自己還留著一塊「下次錄製用」的緩衝**  
   停錄後常見幾十 MB，見 [性能實測「恢復閒置」](/zh-tw/guides/performance-benchmark/)（約 54 MB）。錄過 4K 時，高畫質相關部分大約 **15 分鐘**不用後會逐步釋放。

2. **剛錄過很多路時，數字會暫時接近「錄製高峰」**  
   例如 4～5 路同時錄過，總量兩三百 MB 可能和 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/) 裡的峰值同一量級，**不一定是漏記憶體**。

3. **監控裡的「總記憶體」常把「剛寫過的錄製檔案緩存」也算進去**  
   這是系統為讀寫加速暫存的副本，**不是** bilirec 又多佔了一份程序記憶體；記憶體緊張時系統會自己收回。詳見 [記憶體佔用估算 - 容器總記憶體虛高](/zh-tw/configuration/memory-estimation/#容器總記憶體虛高)。

**你可以：**

1. 對照 [性能實測](/zh-tw/guides/performance-benchmark/)「恢復閒置」— 在附近通常正常。
2. 想再低一點：略減緩衝相關設定（見 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/)）。
3. 若主要是監控「總記憶體」虛高：默認已開啟錄製中定期清理舊檔案緩存；仍偏高時讀 [記憶體佔用估算](/zh-tw/configuration/memory-estimation/)。SSD/HDD 若更在意回放速度，可關閉（見下節）。

**何時要擔心：** 停錄很久仍**持續上漲**、或明顯卡死 — 再查日誌或提 issue；單次高峰後居高不下多數屬於上面 1～3。

### 監控記憶體偏高？兩個相關選項怎麼理解？

**默認已開啟**（一般不用改）：

| 環境變量 | 大白話 | 默認 |
| -------- | ------ | ---- |
| `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS` | **錄製過程中**，每隔一段時間讓系統忘掉「已經寫好、暫時不會再讀」的舊檔案緩存 | `60`（秒） |
| `DROP_FILE_PAGE_CACHE` | **一段錄完或轉碼結束後**，忘掉整份檔案的系統緩存 | `true` |

兩者配合，可避免 Docker / 手機監控裡記憶體隨錄製一路頂到上限。代價是：剛錄完立刻播放、或錄製中回放已寫好的前半段，可能要從磁盤重新讀，略慢。

**冷釋放不是「每 60 秒必清一次」：** 單段檔案要長到約 96 MB 以上才會開始清理舊前綴；短 HLS 分段主要靠關檔時的 `DROP_FILE_PAGE_CACHE`。詳見 [I/O 調優 · 錄製檔案緩存釋放](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android)。

**適合保持默認：** 樹莓派、1GB 容器、Android 後臺、很少錄完馬上拖進度條。

**建議關閉（回放優先）：** SSD / NAS、錄完馬上要秒開播放。設 `DROP_FILE_PAGE_CACHE=false` 且 `LIVE_STREAM_WRITER_COLD_CACHE_RELEASE_SECS=0`。

**和「週期性整檔案落盤」(`LIVE_STREAM_WRITER_SYNC_PERIOD_SECS`) 別搞混：**

- 整檔案落盤：為了**斷電少丟數據**，錄製中反覆把整個檔案刷進卡，**很傷 microSD**，也**壓不住**監控記憶體。
- 上面的 60 秒選項：為了**監控好看**，只清理舊數據，**比整檔案落盤省卡**；關檔時仍會完整保存一次。
- 兩個都設了：以整檔案落盤為準，不再單獨按 60 秒清緩存；**僅當 `COLD_CACHE_RELEASE_SECS>0`** 時，每次 fsync 成功後才順便釋放冷前綴。
- **落盤不會自動釋放緩存**：`SYNC_PERIOD` 負責落盤；釋放緩存靠 fsync 後的冷前綴釋放、或獨立的冷釋放定時器、以及 `DROP_FILE_PAGE_CACHE`，分工不同。

若日誌出現 `釋放檔案頁緩存失敗` 或 `冷緩存釋放失敗`，錄製通常仍可繼續，只是記憶體可能降不下來；見 [I/O 調優 · 錄製檔案緩存釋放](/zh-tw/configuration/io-tuning/#錄製檔案緩存釋放linux--android)。

詳見 [環境變量說明](/zh-tw/configuration/env-reference/) · [I/O 調優](/zh-tw/configuration/io-tuning/)。

<Aside type="tip">
**簡單說：** 停錄後記憶體比剛開機高一些很常見；正在錄的時候若緊張，優先調緩衝設定，別急著減錄製路數。
</Aside>
