## 項目結構

```text
.
├── cmd/
│   ├── backend/          # 後端可執行入口
│   └── androidlib/       # Android c-shared 動態庫
├── internal/
│   ├── controllers/      # HTTP 控制器（auth, convert, file, notify, record, room）
│   ├── modules/          # 核心模塊（bilibili, config, rest）
│   ├── processors/       # 流處理器
│   └── services/         # 業務邏輯（recorder, convert, expose, notify, room, stream...）
├── pkg/                  # 可複用庫（ffmpeg, flv, hls, cloudconvert, pool, swr...）
└── utils/                # 工具函數庫
```

## 錄製流程

1. 通過 `bilibili.Client` 獲取直播流地址（未指定 `stream_profile` 時依次嘗試 HTTP-FLV → HLS-fMP4 → HLS-TS）
2. `stream.Service` 讀取流數據
3. `recorder.Service` 管理錄製任務（自動重連與恢復，恢復時沿用啟動時的畫質與流格式參數）
4. 數據寫入錄製檔案（`.flv`、`.ts` 或 `.fmp4`）

- **HTTP-FLV**：檢測到 PK 等 FLV 檔案頭變更（分辨率切換）時自動分段輪轉
- **HLS-fMP4**：默認將 init segment 與媒體分片順序追加到同一檔案；播放列表重發**內容相同**的 init（如 CDN 序列回退）會跳過並繼續寫入；僅當 init **內容變更**（PK、換源、編解碼參數變化）時才輪轉到新分段，且新檔案以完整 init 開頭，可正常轉 MP4
- **HLS-TS**：不執行檔案輪轉，直播 PK 等不連續性由 HLS 播放列表層自然處理，流中斷時重連繼續寫入同一檔案

首個檔案名形如 `標題-時間戳.ext`，後續分段（HTTP-FLV / HLS-fMP4 輪轉後）會命名為 `標題-時間戳-1.ext`、`標題-時間戳-2.ext`。

## 關鍵特性

- **自動恢復** — 流中斷時自動重連；恢復重試沿用本次錄製的啟動參數（畫質、流格式等）
- **自動分段輪轉（HTTP-FLV / HLS-fMP4）** — FLV 在 PK/分辨率切換時輪轉；HLS-fMP4 在 init 內容變更時輪轉（相同 init 重發不切分），新分段帶完整 init
- **2K/4K 雙池** — 根據 `qn` 自動選用 1080p 或高畫質記憶體池，高畫質池懶加載
- **自動錄製** — 訂閱房間開啟 `auto_record` 後，檢測到開播自動啟動錄製
- **開播檢測** — `subcheck` 分組輪詢已開啟通知或自動錄製的房間，檢測新場次並觸發推送或錄製
- **實時通知** — Web Push + SSE
- **記憶體池** — `pool.BufferPool` 減少 GC 壓力；讀/寫 bytes pool 的空閒複用塊數量由 channel 深度自動推導（約為讀側 channel 的 1/4、寫側的 1/8），在途記憶體公式不變
- **SWR 緩存** — 房間信息 Stale-While-Revalidate（軟 TTL 5 分鐘，硬 TTL 30 分鐘）
- **實時修復** — FLV Tag 時間戳修復與重複 Tag 去重
